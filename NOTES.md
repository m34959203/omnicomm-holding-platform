## 2026-07-01 — Любой временной диапазон мгновенно (Фаза A: on-demand из архива + кэш + прогрев)

**Задача (Дмитрий):** любой выбранный диапазон (10/40/90 дней — что угодно) готов к
запросам и подгружается мгновенно. Данные все в архиве (`raw_store`, агрегаты за год).

**Замер:** сборка снимка из архива БЕЗ Omnicomm ≈2с (10д)…6.7с (90д). Узкое место —
только чтение visits/daily из SQLite; роллап KPI и экономика по 52 узлам — 0.01–0.03с.
Секции текущего состояния (геометрия геозон/Sensor Health/ТО) от периода не зависят →
переиспользуются из последнего снимка. **Кэш-чтение готового снимка — ~25мс.**

**Бэкенд (`api/sync.py`):**
- `build_range_snapshot(start,end)` — собрать снимок за ЛЮБОЙ диапазон из архива (0 обращений
  в Omnicomm): `_reconstruct_base` (tree/vehicle_org/name_map из последнего снимка) → `load_daily`
  +aggregate → `load_visits`→`detect_from_visits(seed=None)` → `_assemble_snapshot(geozones_override=
  base.geozones, sensor/maint из base)`. Кэшируется по period_key.
- `_assemble_snapshot(geozones_override=)` — переиспользовать готовые фичи геозон.
- `prewarm_ranges([дни])` — пред-прогрев трейлинг-окон; вызывается в конце инкрем-синка на `[7,90,180]`.
- `RANGE_MAX_DAYS=400` — потолок (архив ≈год).

**Бэкенд (`api/main.py`):** `_ensure_range_snapshot(period_key)` в `_snapshot` — если ключ-диапазон
(`YYYY-MM-DD_YYYY-MM-DD`) не в кэше → собрать из архива, **single-flight по ключу** (8 табов ждут
одну сборку). Первый заход ~3с, дальше мгновенно. Невалидный/слишком длинный ключ → skip.

**Фронт:** пилюли периода → лестница трейлинг-диапазонов **Сутки/Неделя/Месяц/Квартал/Полгода/Год**
(не «доступные снимки», а вычисляемые ключи; клик → `onRange`→load, бэкенд соберёт если надо) +
**произвольный календарь from–to** в Ribbon (`onRange(`${from}_${to}`)`). Индикатор «сборка…»
(`refreshing`). ВАЖНО: пилюли/календарь больше НЕ запускают полный синк Omnicomm (это делал старый
`syncCustom`→`startSync`); полный забор остался отдельной кнопкой «↻ обновить». SyncBar.tsx — legacy, не используется.

**Проверка:** +5 тестов (`test_range_snapshot.py`), `pytest -q`=**343**, tsc+`pnpm build` зелёные.
Прогрел все 6 пилюль на проде — каждая читается ~25мс, 2004–2006 ТС. Домен 200.
**Фаза B (дневной per-org/veh куб → сабсекунда на ЛЮБОМ впервые запрошенном окне) —
РАССМОТРЕНА И ОТКЛОНЕНА (Дмитрий 01.07).** ROI маргинальный: Фаза A уже делает реальные
сценарии мгновенными (прогретые пилюли + кэш), B ускоряла бы только первый заход РЕДКОГО
произвольного окна (~3с→<1с, один раз, со спиннером «сборка…»). Цена непропорциональна —
инвазивный рефакторинг ядра `data_loader` (числа обязаны совпадать байт-в-байт с дашбордом),

## 2026-07-01 — Авто-сидинг учёток на новые каноничные узлы при синке

Второй слой синхронизации структуры: как только в дереве Omnicomm появляется НОВОЕ
ДЗО/под-ДЗО, ему автоматически заводится логин.

**`api/account_seed.py`** — `seed_new_accounts(kpi_tree)`: для каждого узла без учётки
→ `auth.create_user(slug, token_urlsafe(9), "manager", org_id=узел)`. Слаг —
транслит кириллицы + срез юр-форм (ТОО/АО/СП/НАК…): «ТОО Каратау»→`karatau`,
«ТОО СП ЮГХК»→`yughk`, «АО НАК Казатомпром»→`kazatomprom` (совпадает со старой схемой).
Коллизия имени → суффикс `-<org_id>`. Пароль в открытом виде дописывается строкой в
`data/accounts.xlsx` (лист раздачи, gitignored; `GET /api/accounts` для admin/КАП);
в `users.json` только PBKDF2-хеш.

**Инвариант — аддитивно/идемпотентно:** узел с учёткой пропускается; существующие
учётки/пароли/привязки не трогаются; учётки удалённых узлов НЕ удаляются (остаются,
скоуп пустой — безопасно). Срезы (`/…/`) сюда не доходят (уже выкинуты деревом).

**Хук:** `sync._assemble_snapshot(..., seed_accounts)` после `build_org_kpi_tree`,
в try/except (сидинг не роняет синк). Инкрем-синк → `seed_accounts=True`; полный →
`not demo` (демо-учётки не заводим). Прогресс-строка «Заведено учёток на новые ДЗО: N».

**Проверка:** +6 тестов (`test_account_seed.py`: слаги, per-node, идемпотентность,
только-новые, коллизия-суффикс, xlsx-append). `pytest -q` = **338**. Dry-run на живом
прод-дереве: 0 новых (все 52 узла уже с учётками), слаги совпали. API перезапущен на

## 2026-06-30 — Ускорение бэкфилла треков: трек-ориентированный цикл

Статус синка на момент правки: **агрегаты ~100%** (373 дня · 2004 ТС · 2025-06-23→2026-06-30),
узкое горло — **треки ~39%** по ТС-дням (186k из ~478k; ~71% по глубине календаря, бэкфилл
дошёл назад до 2025-10-10, осталось до 2025-06-23). 6.3 ГБ `data/cache/raw.db`.

**Диагноз:** в `backfill_continuous.sh` каждый цикл (~71 мин) тратил **30 мин (42%) на агрегат-
фазу** (`/api/sync/incremental` за весь год) — холостой ход: год уже залит + догоняется отдельным
cron `0 */3` (`/api/sync/incremental {}`). Хуже — это самый тяжёлый запрос `consolidated`
(`consolidated_s` ~30–37с), бьёт по хрупкой копе `projectkap` против правила «не долбить».

**Фикс (untracked-скрипты `scripts/`):**
- `backfill_trickle.sh` — флаг `SKIP_AGG=1` пропускает агрегат-слайс (оставлен health-gate + треки).
- `backfill_continuous.sh` — трек-ориентированный цикл: треки каждый проход, **полный проход
  (с годовым агрегат-слайсом — страховка от редких дыр) раз в `AGG_EVERY=8` циклов**.

**Эффект:** цикл 71 → ~41 мин, дозалив треков **~5.3 → ~3.3 суток**; нагрузка тяжёлым
`consolidated` на копю ↓ в 8× (в русле feedback-omnicomm-copy-fragile). Параллелизм не трогали
(8 воркеров уже упираются в латентность копи).

**Гоча перезапуска:** старый in-flight trickle держал **унаследованный fd** `flock`
(`/tmp/holding_trickle.lock`), `flock -n` нового цикла падал с кодом 1 → снять надо и детей
(`pkill -P <pid>`), затем поднять через `setsid`. Серверный трек-job в uvicorn резюмируемый —
kill bash-обёртки прогресс не теряет. Cron-дубль `0 */4` по-прежнему гасится тем же flock.

## 2026-06-18 — Milestone 8: переход на БД (SQLite, star schema)

**Сделано (`store.py`, диспетч в `org.py`, `tests/test_store.py` — 7 тестов; полный набор 177 зелёных):**
- `store.py` — SQLite star schema: `dim_org(org_id,name,parent_id,level,type)` +
  `vehicle_org(vehicle_id,org_id)` с индексами. `save/load_org_registry`, `init_db`.
- **Прозрачный диспетч**: `org.save/load_org_registry` смотрят расширение пути —
  `.db/.sqlite`→SQLite (`store`), иначе JSON. Весь стек (CLI `--registry x.db`,
  `holding.build_registry`, портал) работает на БД **без изменений кода**.
- Портал предпочитает `data/org_registry.db`, если есть, иначе JSON.
- Бэкенд изолирован в `store._connect()` — Postgres подключится там же через DSN.
- Тесты: round-trip, перезапись (полная замена), диспетч по расширению (магия `SQLite format 3`),
  чужая/отсутствующая БД → None, build_registry→.db.
- Решение SQLite vs Postgres (§10.9): **SQLite** — ноль инфраструктуры, один файл на dev/сервере/в тестах.

- **РЕШЕНО (§9):** источник данных — **напрямую API Omnicomm** (не БД, не файлы).
  Один аккаунт = один фид КАП. Базис `api_client.py`+`auth.py`. Скрин Lineage от Романа
  больше не нужен — все открытые вопросы по источнику закрыты.
- **Новое требование (§8): row-level контроль доступа.** Конфиденциальность: руководитель
  видит весь КАП, ДЗО видит только своё поддерево, соседи друг друга не видят. В Power BI
  это упиралось в «лицензия на каждого ИЛИ публичная ссылка без конфиденциальности».
  Платформа делает доступ своим слоем (фильтр по поддереву `org_id`) → конфиденциальность +
  экспорт + ноль лицензий разом.
- Уточнена структура: Уранэнерго → {ТФО, ШФО}; УМЗ → {УМЗ Курчатов}.

## Чек-поинт 2026-06-22 (5) — детекция сбоев датчиков (уточнения Антона)

Антон уточнил по индикации сбоев датчиков → новая способность платформы, `docs/knowledge-base/10`:
- Прямой индикации сбоя ДУТ в Omnicomm нет (аналитик видит по «обрыву» графика топлива). Источник
  авто-детекции — отчёт **«Журнал»** (сырьё по всем узлам с сетевым адресом): пропали данные по
  адресу ДУТ → сбой ДУТ. «Светофор» над списком техники = давность данных ТЕРМИНАЛА, не датчика.
- Модель двухуровневая: уровень 1 терминал (receive_data/светофор) развязать с уровнем 2 датчик
  (по адресу из «Журнала», при онлайн-терминале). Omnicomm уже даёт статусы LLS (Готов/Нет/Ошибка)
  и событие «Сбой LLS» — поднять авто + распространить на все датчики. Это вентиль качества ПЕРЕД
  KPI/роллапами (битый ДУТ → топливо ТС невалидно → исключать из рейтингов).
- CAN-подключение: «крокодилы» (бесконтактный, рекоменд. вендора) / CAN-LOG / прямое → внёс в каталог 01.
- Открытый 🔴: доступен ли «Журнал» (сырьё по узлам) через REST API — если нет, только терминальный
  уровень. Запросить «Руководство по администрированию» Omnicomm + пример выгрузки «Журнала».

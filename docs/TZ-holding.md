## 1. Роли и контроль доступа

- **Две роли, без тонкого RBAC** (решение заказчика): `начальник` (весь холдинг) и `ДЗО` (только своё поддерево `dim_org`).
- **Row-level доступ по поддереву** — жёсткое требование: ДЗО видит свою технику + под-ДЗО + подрядчиков, не видит соседей; начальник — весь КАП.
- **Одинаковый дашборд** для всех (просмотр + экспорт), различие только в видимом scope. Никакого редактирования данных.
- **Multi-tenant:** у каждого ДЗО свой логин/пароль. ✅ в коде (`auth.py` + `org_id`), нужны учётки по ДЗО при онбординге.
- Роль «ответственный за ТО» (подтверждение ТО) — в рамках scope ДЗО (см. §6). ⚠️ Подтверждение ТО — это **запись**, что конфликтует с «только просмотр»; решить как микро-роль/отдельный мини-флоу (вопрос к Диме/Антону, [REVIEW](REVIEW-2026-06-22.md)).

## 8. Нефункциональные требования

- **NF1 Безопасность/ПДн:** креды только из ENV; данные в РК; без трансграничной передачи; пароли клиентов — шифрование (не base64 в проде, остаётся к онбордингу). 🟡 добавлены: глобальный rate-limiter ≤170/мин на аккаунт (защита сервера Omnicomm при лимите 180/мин) + TTL-кэш карточки ТС; вентиль доверия к данным (`classify.py`, фильтр не-ТС из рейтингов).
- **NF2 Доступ:** fail-closed изоляция ДЗО (перепроверка scope перед рендером).
- **NF3 Масштаб:** ~2000 ТС, 23 ДЗО; роллапы и дашборды без деградации; SQLite сейчас, Postgres при росте.
- **NF4 Надёжность сбора:** батчинг ≤50 ТС / ≤31 дня, ретраи, упреждающий refresh JWT, обработка кодов ошибок Omnicomm.
- **NF5 Локализация:** RU/KK; денежные величины в ₸ (млн/тыс).
- **NF6 Доставка:** веб-портал + Excel; мобильный просмотр.

## 9. MVP vs Дальше (скорректировано аудитом 22.06 — [REVIEW](REVIEW-2026-06-22.md))

> Вывод аудита: исходный MVP (5 пунктов директора) — это **упаковка доставки**, не дифференциатор; ни один пункт не отличает от Power BI. Поэтому MVP ведём двумя треками параллельно.

**MVP-доставка (🔵 директор, по важности):** (1) Excel-экспорт → (2) два языка RU/KK → (3) мобильная версия → (4) логин на ДЗО [есть] → (5) настраиваемая частота обновления (админ-настройка, не пользовательская). Плюс база: иерархия + дашборды ДЗО + экономика (есть).

**MVP-доказательство ценности (добавлено аудитом):** хотя бы ОДИН срез уникальной ценности на реальных данных КАП к той же демонстрации — приоритетно **скоростной режим по СТ КАП → дисциплинарка/штраф** (данные уже есть). Плюс **терминальный Sensor Health (R7.1)** — как инфраструктура доверия к цифрам (без неё деньги оспорят на первой встрече).

**Дальше (ядро ценности):** полный `geozone_limit` + детекция превышений (ночной батч на сабсете) → движок рекомендаций на букве закона (с инвариантами [kb-05 §0](knowledge-base/05-recommendation-engine.md)) → контроль ТО → реестр датчиков.

**До прод-онбординга КАП (P0 безопасность/процесс):** криптостойкое хранение паролей (замена base64); процесс онбординга ДЗО + приёмочный тест изоляции с участием самого ДЗО; КАП Business Case (плательщик/цена/ROI).

## 10. Критерии приёмки (выборка)

- ДЗО входит под своим логином и видит **только свою** технику (row-level); начальник — весь КАП. ✅
- Дашборд ДЗО выгружается в Excel одной кнопкой; интерфейс доступен на RU и KK; читается с телефона. ✅
- Превышение на участке квалифицируется по СТ КАП (лимит геозоны × тип ТС) и по КоАП РК с суммой штрафа. ✅ (для дорог общего пользования; на техдорогах — дисциплинарка)
- ТС с битым датчиком топлива помечен и не искажает топливные рейтинги. ✅
- Контроль ТО: наработка считается из фида, алерт срабатывает, подтверждение сбрасывает цикл. 🟡 (наработка+алерт есть; цикл подтверждения — write-flow впереди)

## 11. Ограничения и допущения

- **Видео/ADAS/DSM** — отдельный контур интегратора, в Omnicomm не передаётся; в платформе — потенциальный доп. источник, не автоматический фид.
- **«Журнал» (сырьё по датчикам) через REST API недоступен** на текущей учётке (проверено) → Sensor Health пока терминального уровня; для полного сенсор-уровня нужен расширенный доступ/«Руководство по администрированию» Omnicomm. **Частичное снятие:** напряжение бортсети доступно через `/vehicles/{id}/state` (показано в карточке ТС).
- **Серийники датчиков** в фиде не приходят → реестр ведётся вручную из монтажных отчётов.
- **Контроль ТО «от нуля»** — точность растёт по мере накопления; прошлые ТО неизвестны без данных заказчика (нужен T0-offset при онбординге).
- **Справочник геозон загружается только как verified-артефакт** (ручное ревью заказчика, чистка аномалий — Заречное/дубли 40-80/опечатки, [09 §5](knowledge-base/09-kap-geozone-speed-standard.md)); не авто-парс из имени.
- **Досье водителя (R5.3) — только при правовом основании** (согласие/локальный акт), минимизации полей, без авто-решений по возрасту (ПДн РК; [REVIEW](REVIEW-2026-06-22.md) R-INV-7).
- **`online.omnicomm.ru` — РФ-домен** → подтвердить геолокацию копии/каналов перед заявлением «без трансграничной передачи ПДн».
- Профильные системы учёта Казатомпрома недоступны.

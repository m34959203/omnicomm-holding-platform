# IA-спека: дашборд ДЗО

Информационная архитектура дашборда организации (ДЗО / под-ДЗО / холдинг) в
holding-портале. Цель документа — задать **что и в каком порядке** показывать,
чтобы было что реализовывать в `dashboard.py` / `holding_app.py`, а не только
принципы. Все блоки привязаны к реальным полям модели.

## 1. Job-to-be-done и аудитория

- **Задача:** «где по этому ДЗО за период течёт ₸ и что с этим делать», а НЕ
  «где машина сейчас». Это управленческо-финансовый слой, не диспетчерская.
- **Аудитория:** глава ДЗО, аналитик КАП (Саша), руководство холдинга. Считываемость
  **< 10 секунд** до первого вывода. Ясность важнее плотности.
- **Период, не реалтайм.** Один прогон = один период (`ReportPeriod`).

## 2. Принципы (что взяли у DMMS, что отвергли)

Взято (см. разбор DMMS): доменные блоки телеметрии вместо свалки метрик;
push-лента аномалий вместо «иди ищи»; entity-drill (ДЗО → ТС); «одна модель —
разные высоты» (холдинг/ДЗО/под-ДЗО — один макет, меняется только уровень роллапа).

Отвергнуто: **карта как центр композиции** (это операционная парадигма под их
железо; у нас ось — деньги и ДЗО, карта — максимум виджет); **плотность
диспетчерской видеостены** (наша аудитория — не обученный оператор); **реалтайм**.

Бизнес-инварианты (наследуются из fleet-report, НЕ нарушать):
- Без красного и обвинительных формулировок. Статусы: зелёный «лучше» / янтарь
  «внимание» / серый «нейтрально». Аномалии — только «требует проверки».
- Столбец «возможные сливы топлива» НИКОГДА не в выдаче.
- Перерасход не утверждать без согласованных норм (`norm_coeff`/нормы заданы).
- В каждой денежной строке — тег метода: `факт` / `норма` / `≈ оценка`.
  «≈» только на оценочных (`Bucket.is_estimate=True`), измеренные — без «≈».
- Светлый корпоративный стиль, читаемый текст.

## 4. Блоки и привязка к данным

Источник для всего — `report = dashboard.build_org_report(org_id, …)` (тип
`FleetReport`) + `eco = economics.build_economics(report)` (тип `Economics`) +
`OrgKPI` из `rollup.build_org_kpi_tree` для Z7. `report.kpi` — `FleetKPI`.

### Z0 — Контекст-бар
- Хлебные крошки по `OrgTree.ancestors(org_id)` → имена узлов (КАП › … › текущий).
- `report.client_name`, `report.period.human()`.
- ТС: `OrgKPI.vehicle_count` (всё поддерево) и `direct_vehicle_count` (прямо
  привязанные). Покрытие данными: `kpi.vehicles_with_data / kpi.vehicles_total`.
- `report.season` (Лето/Зима — влияет на нормы), `report.generated_at`.

### Z1 — North-star KPI (4 плитки, сериф-числа, дельта ▲▼)
Дельты — из `report.trends` (% к `previous_kpi`), цвет-статус через хелпер
`_kpi_delta` (зелёный/янтарь/серый, БЕЗ красного). Только при наличии
`previous_kpi`, иначе плитка без дельты.

1. **Затраты на топливо** — `kpi.total_fuel_cost` ₸. Тег `факт`.
2. **Адресуемые потери** — `eco.total_existing_kzt` (или `eco.coi_monthly_kzt`
   с пометкой ₸/мес). Тег `факт`+`≈оценка` смешанный → пометить «вкл. оценки».
3. **Холостой ход** — `kpi.idle_hours_share` (%). Тег `факт`.
4. **Экономия** — если `report.savings` непустой: накопленный счётчик; иначе
   **перерасход к нормам** `kpi.total_overrun_cost` (только если нормы заданы),
   иначе плитка скрыта. Тег `норма` / `измерено`.

### Z2 — Лента внимания (push, DMMS-паттерн 2, наша палитра)
- Источник: `report.alerts` (строки из `alerts.py`). Топ 5–7, остальное в
  экспандер «ещё N».
- Каждый пункт: иконка-статус (янтарь «внимание» / серый «инфо», без красного),
  текст, и ₸ если есть. Без обвинений — формулировка «требует проверки».
- Пусто → «Сигналов за период нет» (это норма, не дефект).

### Z3 — Где деньги (доменный блок, ранжированные утечки)
- Источник: `eco.buckets` (`Bucket`: `label`, `existing_kzt`, `potential_kzt`,
  `is_estimate`, `note`). Сортировка по `existing_kzt` убыв.
- В каждой строке: метод-тег (`is_estimate` → `≈ оценка`, иначе `факт`/`норма`)
  + `note` мелким. Срезы НЕ суммируются в одну цифру (показать раздельно).
- Подвал: `eco.coi_monthly_kzt` / `coi_annual_kzt` с пометкой «оценка,
  экстраполяция» (защита экстраполяции на коротком периоде уже в economics).
- Если `kpi.fuel_price_kzt == 0` → весь Z3 скрыт (без цены денег нет).

### Z4 — Доменные блоки телеметрии (DMMS-паттерн 3, именованные карточки)
Не свалка метрик — 3–4 карточки по смыслу. Учитывать `mobile_count` /
`stationary_count` (показывать релевантные метрики каждому классу).

- **Топливо:** `total_fuel_l`, `mobile_fuel_per_100km` (л/100км мобильных),
  `fuel_cost_per_km`.
- **Моточасы / двигатель:** `total_engine_hours`, сплит `movement_hours` vs
  `total_idle_hours` (+ `idle_hours_share`), `weighted_fuel_per_motorhour`
  (спецтехника). Bullet «факт/цель/среднее» для холостого хода.
- **Использование:** `utilization_calendar`; если `time_fund_hours_per_day>0` —
  `utilization_fund` (>1 = «сверх фонда», пометить, не как ошибку).
- **Вывоз (опц.):** только если `haul_volume_m3>0` — `fuel_cost_per_m3` ₸/м³.

### Z5 — Первоочередные ТС (entity-drill, DMMS-паттерн 4)
- Источник: `eco.worst_vehicles` [(имя, ₸)] + `report.scorecard` (балл
  проблемности). Топ-5, ранжировано по ₸-потенциалу.
- Клик по строке → карточка ТС: его топливо/моточасы/холостой ход/перерасход.
- Тон нейтральный: «потенциал улучшения», не «нарушитель». `severity=REVIEW`.

### Z6 — Счётчик подтверждённой экономии (условный)
- Только если `report.savings` непуст (baseline заморожен). Накопленный итог +
  step-серия. Иначе блок-заглушка «Baseline не заморожен — заморозить за период».
- Baseline из 1 периода → пометка «предварительно» (методология IPMVP-lite).

### Z7 — Структурный подвал
- **Состав парка:** пончик `mobile_count` vs `stationary_count`.
- **По под-ДЗО (мульти-высота):** если `OrgKPI.children` непуст — мини-таблица
  роллапа по дочерним узлам (имя · ТС · топливо ₸ · холостой ход %), каждая
  строка кликабельна → переход на дашборд дочернего узла. Это и есть паттерн
  «одна модель — разные высоты»: лист-ДЗО этот блок не показывает.
- **Карта (опц., НЕ центр):** маленький виджет, только если есть трек; по
  умолчанию свёрнут. Никогда не главный объект экрана.

## 5. Сквозное поведение

- **Row-level доступ:** дашборд доступен только если `org_id ∈
  viewer.visible_scope` (`OrgTree.visible_scope`, fail-closed). Z7-дети — только
  внутри поддерева зрителя. Перепроверять перед рендером (defense-in-depth, как
  в `holding_app.py`).
- **Мульти-высота:** один макет на холдинг (КАП) / ДЗО / под-ДЗО — меняется
  только уровень роллапа (`rollup`), Z7-таблица детей появляется лишь у узлов с
  children. Холдинг-уровень = тот же дашборд для `org_id="holding"`.
- **Экспорт:** `render_org_report` → PDF / PPTX / Excel (escape hatch отчёта).
- **Порядок чтения:** Z1 (что важно) + Z2 (что горит) — верх; детализация ниже.

## 6. Сознательно НЕ делаем
Карта-центр · плотность диспетчерской · реалтайм-мониторинг · live-телеметрия ·
красный цвет/обвинения · столбец сливов · перерасход без норм.

## 7. Что для этого нужно в коде (ориентир, не обязательство)
- `dashboard.build_org_report` уже отдаёт `FleetReport`; для Z3 вызвать
  `economics.build_economics(report)`; для Z7 — `rollup.build_org_kpi_tree`.
- Рендер Z0–Z7 — в `holding_app.py` (после «Сформировать дашборд») и/или в
  `dashboard.render_org_report` для PDF/PPTX-зеркала.
- Хелпер `_kpi_delta` (цвет-статус дельты) переносится из fleet-report.

## 1. Что за проект

Holding-scale аналитическая платформа автопарка для холдинга **КАП** (вероятно
Казатомпром). Данные тянутся **напрямую с API Omnicomm** (один аккаунт = один фид),
нормализуются в иерархию организаций, переводятся в деньги (₸) и раздаются как
дашборды/отчёты на каждое ДЗО — **без посемест­ных лицензий Power BI**.

Выросла из single-client проекта `omnicomm-fleet-report` (его движок переиспользуется),
поверх достроен **holding-слой**.

## 2. Контекст заказчика

- Центральная учётка холдинга (**КАП**) → **23 ДЗО** → под-ДЗО → подрядчики.
  Структура реальных под-ДЗО: Уранэнерго → {ТФО, ШФО}; УМЗ → {УМЗ Курчатов}.
  **У одного ДЗО может быть несколько подорганизаций** (fan-out любой).
- Парк ~1427 ТС (1035 активны за 7 дней на момент осмотра).
- Текущая витрина — Power BI «Мониторинг скоростного режима-4» (только скорость).
  Её боли: нет топлива/экономики, нет моточасов/акселерометра, доставка через
  посемест­ные лицензии (~8500 ₸/мес × каждый) + ручная выгрузка по воскресеньям,
  «колонка-на-ДЗО» не масштабируется.
- Power BI трогать **только визуально** (правило заказчика: ничего не редактировать/сохранять).

## 3. Зафиксированные решения (НЕ переоткрывать)

| Вопрос | Решение |
|---|---|
| Источник данных | **Напрямую API Omnicomm** (не БД, не файлы). Базис `api_client.py` |
| Подрядчики/под-ДЗО | Все в **одном фиде КАП** → узлы `dim_org`, мульти-аккаунт НЕ нужен |
| Модель организаций | Иерархия `dim_org` (звезда), а не «колонка-на-ДЗО» |
| Контроль доступа | **Row-level по поддереву**: ДЗО видит свои под-ДЗО+подрядчиков, не соседей; корень=весь КАП |
| Источник иерархии | Берём из **дерева ТС Omnicomm** (узел дерева = организация) |
| Чистка выбросов скорости | По **физике движения** (расстояние/Δt), помечать не удалять — *спроектировано, не реализовано* |
| Хранилище | Пока JSON (как `clients.py`); БД (SQLite/Postgres) — отложено (§10.9) |

## 4. Карта модулей

**Holding-слой (новое в этой платформе):**
- `org.py` — `dim_org`: `Org`/`OrgTree`, построение из дерева Omnicomm
  (`build_from_omnicomm_tree`), доступ по поддереву (`can_view`/`visible_scope`),
  `assign_org_ids` (ингест), `filter_vehicles_for_viewer` (fail-closed), JSON-реестр.
- `rollup.py` — роллапы KPI по иерархии: `rollup_kpi` (org_id→FleetKPI по ТС поддерева),
  `build_org_kpi_tree` (дерево `OrgKPI`). Переиспользует `analytics.compute_kpi`.
- `dashboard.py` — дашборд на ДЗО: `build_org_report`, `render_org_report`,
  `render_for_scope` (рендер в пределах scope, изоляция ДЗО).
- `holding.py` — **end-to-end оркестратор**: `build_registry`, `run` (реестр→ингест→
  роллапы→дашборды), `fetch_fleet`/`run_from_client` (забор из живого Omnicomm). `HoldingRun`.
- `auth.py` (расширен) — у пользователя `org_id`; `authenticate`/`user_org`/`get_user`.
- `models.VehicleMetrics.org_id` — привязка ТС к узлу.

**Унаследованный движок (single-client, переиспользуется как есть):**
- `api_client.py`/`auth`(Omnicomm JWT в api_client)/`data_loader.py` — забор из Omnicomm/Excel.
- `validator.py` — чистка/аномалии (порог `MAX_PLAUSIBLE_SPEED_KMH`).
- `analytics.py` — `compute_kpi`, `analyze` → `FleetReport`.
- `economics`/`norms`/`savings`/`fuel_price` — топливо→₸, перерасход.
- `vehicle_types.py` — типы ТС + профили. `loading.py` — моточасы спецтехники.
- `charts.py`/`report_builder.py` — графики + `.pptx`/`.html`. `scheduler`/`mailer` — доставка.

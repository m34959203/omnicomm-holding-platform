## 1. Модель доступа — РЕШЕНО (уточняет прежнее «row-level = жёсткое требование»)

**Что сохраняется:** row-level доступ по поддереву `dim_org` — главное требование. Начальник КАП видит **всё**; каждое ДЗО — **только свою технику** (своё поддерево), соседей не видит.

**Что упрощается (вводная директора):** **тонкое разграничение ролей НЕ нужно.** Все сотрудники Казатомпрома получают **одинаковый дашборд с одинаковыми правами** (просмотр + экспорт), «чтобы не феерили». Никаких editor/менеджерских функций редактирования данных — смотреть свою технику и выгружать, и всё.

**Multi-tenant по ДЗО (опыт Саши с Power BI):** был общий дашборд, который видел только один начальник, а у **каждого ДЗО — свой логин/пароль** и доступ к своим данным. Прогноз Саши: как только общий продукт будет готов, Волковгеология захочет видеть свою технику под своей учёткой, остальные ДЗО попросят то же. → закладываем **отдельные учётки на ДЗО** сразу.

**Как это ложится на код (`auth.py` — уже почти готово):**
- `admin` — начальник холдинга, `org_id = None` → весь КАП.
- `manager` — ДЗО, `org_id = <узел dim_org>` → только своё поддерево (scope-изоляция `visible_scope`).
- Один и тот же дашборд `holding_app.py`; различие — только в видимом scope. Тонких ролей/прав сверх этого не вводить.
- **Не усложнять RBAC.** Две роли (начальник / ДЗО) = достаточно.

## 2. MVP-приоритеты — список директора (по важности)

| # | Требование | Статус в коде | Что доделать |
|---|---|---|---|
| 1 | **Выгрузка в Excel** (самое главное) | `--xlsx` есть в CLI (очищенный Excel) | Кнопка экспорта в портале `holding_app.py` на дашборде ДЗО (выгрузка текущего среза) |
| 2 | **Два языка (RU / KK)** | нет в портале | i18n-слой портала и отчётов (паттерн двуязычия — как в других проектах) |
| 3 | **Мобильная версия** | Streamlit адаптивен частично | Проверить/довести мобильный вид портала (телефон) |
| 4 | **Свой логин/пароль на ДЗО** | есть (`auth` + `org_id`) | Завести учётки по ДЗО при онбординге |
| 5 | **Настраиваемая частота обновления** | `scheduler.py` есть | Параметр «сколько раз в сутки тянуть данные» на уровне портала/деплоя |

> Эти пять — must-have для продукта КАП. Остальное (доменная телеметрия, рекомендации) — поверх.

## 3. Ограничения интеграции — ВАЖНО (вводная директора, 18.05)

- **GPS-мониторинг = Omnicomm.** Подавляющее большинство оборудования парка — производства Omnicomm; стороннего немного. → наш API-контур покрывает основную телеметрию (топливо/обороты/трек — подтверждено смоуком).
- **Видеотерминалы — НЕ Omnicomm и НЕ передают данные в Omnicomm напрямую** (своё ПО, отдельный контур). → **видео / ADAS / DSM не придут через Omnicomm API.** Это уточняет «честную границу»:
  - поведенческие метрики (усталость/отвлечение/телефон) и видео — **отдельная интеграция/контур**, не автоматический фид Omnicomm;
  - рекомендация «дооснастить DSM/ADAS» (лестница эскалации, [05](knowledge-base/05-recommendation-engine.md)) = **продажа отдельной системы**, данные которой подключаются отдельно (или живут в своём ПО). Это и есть upsell, а не «включить галочку».
  - на сводном API копии КАП `uniDataList` (надстройка) был пуст — согласуется.
- **Профильные системы учёта Казатомпрома недоступны** — туда доступа не дают. Не закладываемся на их данные.

## 4. Бизнес-рамка (вводные 18.09–18.10)

- Доступ к внутренним делам ДЗО нам не нужен. Вопрос продукта: **что мы устанавливаем, какой мониторинг показываем и на что можем «раскрутить» (upsell)** — ровно лестница эскалации из [05](knowledge-base/05-recommendation-engine.md).

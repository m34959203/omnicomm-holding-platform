# Handoff 2026-06-06 №1 — ревью отчёта, UX/UI и функционала

Кому: Cowork. Задача: независимая оценка системы — (1) качество отчёта,
(2) прогон по UX/UI платформы, (3) сверка с функциональными требованиями.

## Что это за система
Python-платформа автоанализа автопарка из Omnicomm Online → клиентский отчёт
(.pptx/.html/.xlsx) с перерасходом/экономией в ₸. Мульти-клиент (Streamlit),
встроенный планировщик авторассылки, модуль «Работа на погрузке» для мусоровозов.
Полное описание — `docs/platform.md`, гайд — `CLAUDE.md`, обзор — `README.md`.

## Как запустить локально
```bash
pip install -r requirements.txt
# демо-отчёт (без боевых данных):
python -m omnicomm_report --source excel --input samples/fleet_sample.xlsx \
    --client "Демо" --fuel-price 320 --html --xlsx
# платформа:
streamlit run app.py     # боковая навигация: Отчёты / Парк и клиенты / Автоматизация
```
Боевой контур (Горкомтранс) — учётка в `.env` (gitignored), не в репо. Живой
пример отчёта по боевым данным: `output/scheduled/горкомтранс/report_*.html`
(собран планировщиком; открой в браузере).

## Что оценить

### 1. Отчёт (деливерабл)
Разделы (pptx+html): титул+KPI, ключевые выводы, классификация техники, профиль
парка, пробег, топл. эффективность, использование/холостой ход, расход без
движения, финансовая оценка (₸), работа на погрузке (л/моточас + GPS-карта),
перерасход/экономия по нормам, сигналы/приоритеты, потенциал экономии (что-если +
бенчмарк), операционные отклонения, рекомендации, методология.
Оцени: полноту, читаемость для руководителя, корректность формулировок
(инвариант: сливы не показываем, аномалии — «требует проверки»), визуал графиков.

### 2. UX/UI платформы (`app.py`, Streamlit)
Разделы навигации: 📊 Отчёты (по клиенту / из файла) · 🚛 Парк и клиенты
(Парк техники, Шаблоны техники, Новый клиент) · ⏱ Автоматизация (Планировщик,
Журнал действий). Оцени: логичность навигации, понятность форм (ввод учётки,
период пресетами, цена топлива + сверка с поставщиком, e-mail-рассылка,
редактор паспортов/норм, редактор шаблонов типов), наглядность статусов,
тексты/подсказки, узкие места и предложения по улучшению.

### 3. Функциональные требования
Сверь реализацию с ожиданиями заказчика (Горкомтранс, вывоз ТКО):
- мусоровоз: расход по моточасам, не только л/100 км (дуальная норма) — есть;
- нормы по двигателю (паспорт/замер) → перерасход/экономия в ₸ — есть;
- классификация техники управляет расчётом — есть;
- сезон лето/зима → коэффициент к норме (влияет на сумму) — есть;
- сверка/автоподстановка цены ГСМ с поставщика — есть;
- авторассылка отчётов/сигналов по расписанию (демон) — есть;
- аудит действий (особенно шаблоны/нормы) — есть.
Что НЕ покрыто (нужны данные/оснащение): датчики надстройки (точное разделение
погрузка/простой), геозоны контейнерных площадок, ₸/контейнер/тонна, driver
behavior. Оцени, насколько критичны эти пробелы и что приоритизировать.

## 1. Проблема

Холдинг КАП эксплуатирует **~2000 единиц техники** в **23 ДЗО** (рудники, логистика, спецтехника, легковой и вахтовый транспорт) на телематике **Omnicomm Online**. Текущая управленческая витрина — Power BI «Мониторинг скоростного режима» (~4 страницы, только скорость). Её ограничения:

- **Только безопасность/скорость** — нет топливной и экономической аналитики (перерасход в ₸), нет КПД спецтехники (моточасы, простой под нагрузкой), нет контроля ТО и качества данных.
- **Доставка дорогая и ручная** — посемест­ные лицензии Power BI (≈8500 ₸/мес × получатель) + ручная выгрузка каждому ДЗО раз в неделю.
- **Модель не масштабируется** — «колонка на каждое ДЗО» ломается при добавлении ДЗО/под-ДЗО/подрядчика.
- **Рекомендации — «ИИ-вода»** — встроенный «ИИ-помощник» выдаёт общий план без привязки к статье закона, сумме штрафа и конкретному действию (прямая оценка директора по развитию Omnicomm).

Параллельно у холдинга есть **СТ Казатомпром** — стандарт безопасной эксплуатации GPS с матрицей скоростных лимитов по зонам риска × типам ТС и справочником ~400 именованных геозон. Сейчас он живёт на бумаге и в настройках Omnicomm, но не превращён в управленческую аналитику нарушений и ущерба.

## 2. Видение

**Единая аналитическая платформа поверх Omnicomm** — «мозг и память холдинга»: собирает телеметрию по всем ДЗО через API, нормализует в иерархию организаций (Холдинг → ДЗО → под-ДЗО → подрядчик), переводит в деньги (₸) и **раздаёт каждому ДЗО свой дашборд и отчёты — без посемест­ных лицензий**, с доступом строго к своей технике.

Платформа **не переизобретает** то, что уже считает Omnicomm (расход, заправки/сливы, моточасы, SafeDrive), и **не лезет в видео-железо** (это отдельный контур интегратора). Её ценность — слой, которого нет ни у Omnicomm, ни у Power BI, ни у видеоинтегратора: **кросс-ДЗО агрегация + экономика + рекомендации на букве закона + контроль ТО и качества данных**.

## 3. Ценностное предложение

| Пласт | Что даёт | Кому больно сейчас |
|---|---|---|
| **Экономика топлива** | Перерасход/экономия в ₸, нормы, рейтинги ДЗО | нет в Power BI |
| **Скоростной режим на букве закона** | Превышения по геозоне × типу ТС (СТ КАП), квалификация по КоАП РК, сумма штрафов, вероятный ущерб, действие | Power BI показывает факт, не деньги/закон |
| **КПД спецтехники** | Моточасы, простой под нагрузкой, ₸/моточас (буровые, погрузка) | не считает никто |
| **Контроль ТО** | Авто-учёт наработки, заблаговременный алерт, подтверждение цикла | ведётся вручную/никак |
| **Качество данных (Sensor Health)** | «Светофор» терминалов, пометка ТС с битыми датчиками → не врать в KPI | аналитик ловит глазами по графику |
| **Доставка** | Дашборд на ДЗО + Excel + 2 языка + мобильная версия, без лицензий | посемест­ные лицензии + ручной экспорт |

**Главный дифференциатор — рекомендации на букве закона:** нарушение → статья КоАП РК → частота → сумма штрафов (МРП × ₸) → вероятный ущерб → конкретное действие (беседа с водителем; при систематике — дооснащение DSM/ADAS = инструмент продаж интегратора).

## 4. Позиционирование (с кем не конкурируем)

- **vs Omnicomm Online** — мы надстройка, не замена. Omnicomm остаётся источником телеметрии и считает per-ТС; мы агрегируем по холдингу и переводим в управленческие решения.
- **vs Power BI** — заменяем его как витрину: дешевле (без лицензий на получателя), масштабируемо (иерархия), богаче (деньги/закон/ТО).
- **vs видеоинтегратор IT-Transport (ИВТ)** — у них «глаза и нервы» (DMS/ADAS/MDVR, отдельный контур, в Omnicomm не передаёт). У них **нет** ИИ-аналитики, скоринга и холдингового среза — это наш участок. Их события — потенциальный доп. источник для нас.

## 5. Бизнес-модель

- «Аналитика как услуга» поверх существующей телеметрии холдинга — без продажи железа.
- **Upsell-воронка:** платформа автоматически обосновывает дооснащение (систематические нарушения → DSM/ADAS), контроль ТО и качества данных — как сервисные линии.
- Защита данных: вся инфраструктура в РК, без трансграничной передачи ПДн (требование заказчика).

## 6. Границы (scope)

**Делаем:** сбор из Omnicomm API, иерархия организаций, дашборды/отчёты на ДЗО, экономика, скоростной режим по СТ КАП, рекомендации на букве закона, контроль ТО, терминальный Sensor Health, экспорт Excel, 2 языка, мобильный вид.

**Не делаем:** видео-железо и видеоаналитику (контур интегратора); не переизобретаем расчёты Omnicomm; не лезем в профильные системы учёта Казатомпрома (доступа нет); не строим сенсор-уровневую детекцию сбоев, пока нет доступа к «Журналу» через API (см. [Техническое решение](SOLUTION.md) §Ограничения).

## 7. Стейкхолдеры

- **Заказчик:** руководство КАП (видит весь холдинг) + ДЗО (видят свою технику).
- **Аналитик заказчика (Саша):** ведёт текущую витрину, формулирует требования multi-tenant.
- **Директор по развитию Omnicomm:** задаёт принцип «на букве закона» и upsell-логику.
- **Интегратор (Антон / IT-Transport):** телематика, монтаж, видео-контур, доступ к данным.
- **Данияр:** список опасных грузов (ADR). **Дима, Антон:** согласование функционала.

## 8. Текущий статус (на 2026-06-23)

**Платформа в проде: `omnicomm.technokod.kz`** на боевой копии КАП (`projectkap @ online.omnicomm.ru`, ~1962 ТС, 73 узла оргструктуры, 274 теста). Стек переехал с Streamlit на продуктовую тройку: **Python-движки → FastAPI-мост (`api/`) → Next.js фронт (`web/`)** за реверс-прокси; чтения отдаются **мгновенно из кэш-снапшота** и не ходят в Omnicomm.

**Что в проде (5 доменов аналитики):**

- **Executive-дашборд:** scope-rail (дерево + поиск, row-level доступ), 6 health-KPI, лента «что требует внимания» (кликабельна — ведёт к сути), обзор по доменам и домен-табы. Графики — инлайн-SVG.
- **Деньги** — экономика топлива в ₸, COI (cost of inaction).
- **Скоростной режим СТ КАП** — детекция по визитам геозон (430 ТС), рекомендации «на букве закона». **Сверка ст. 592 КоАП на adilet выполнена** (ред. Закон РК 03.10.2024 №131-VIII): суммы штрафов показываются для дорог общего пользования (на техдорогах — дисциплинарка СТ КАП); рекомендации структурированы (Нарушение → Частота → Квалификация → Штраф → Ущерб → Действие).
- **Геозоны** — Яндекс-карта ~199 зон.
- **Качество данных** — Sensor Health (терминальный уровень).
- **Контроль ТО** — наработка + статусы.
- **Карточка ТС:** трек на спутнике (Яндекс) + график скорости + телеметрия + напряжение бортсети.
- **Доставка (MVP):** Excel, RU/KK, мобильный вид, выбор снимка + произвольный период по календарю. Тема Omnicomm Online (цвета + Roboto) + переключатель светлая/тёмная.
- **Вентиль доверия к данным** — фильтр не-ТС (АЗС, ёмкости, ФЭС, генераторы, IMEI) из денежных рейтингов.
- **Защита сервера Omnicomm** — глобальный rate-limiter ≤170/мин на аккаунт (лимит Omnicomm 180/мин) + TTL-кэш карточки ТС. Архитектурный принцип: платформа = единственный буфер; раздача ДЗО-учёток не множит нагрузку на Omnicomm (читают из кэша).

**Снято с прежних ограничений:** напряжение бортсети найдено в `GET /ls/api/v1/vehicles/{id}/state` (не в consolidatedReport) — частично закрывает «сенсор-уровень недоступен по REST» для напряжения/уровня 1.5 Sensor Health.

Дальше — детекция превышений по сырому треку (точка→геозона) вместо визитов; накопление наработки ТО между синками; гарды экономики COI на коротких окнах. Детали и ограничения — [SOLUTION §6](SOLUTION.md).

## Фаза 2 — Ядро ценности: скоростной режим «на букве закона»

- ⛔ **Seed-датасет геозон** из СТ КАП (~400, имя→ДЗО→лимит→тип) — verified-артефакт с чисткой аномалий (Заречное с чужими именами, дубли 40/80, опечатки, СТОП=0). [P0-B]
- 🟡 **`geozone_limit(геозона, тип_ТС)`** — ✅ движок `geozones.py` (матрица 6×3, классификация риск-зоны по имени, гейт «КоАП vs дисциплинарка», дефолтный классификатор категории ТС, именованный лимит = источник истины, парсинг «NN км/ч»). 10 тестов; проверен на 200 боевых геозонах. ⛔ осталось: полный seed-справочник (~400 строк) + матчинг к Omnicomm по `(owner_org+норм.имя)`.
- ⛔ **Детекция превышений по треку** — **ночной батч на сабсете** (триаж по `mv.maxSpeed`, не по всему парку 8×/сутки); по **устойчивому сегменту** + фильтр ускорения; нарезка периода в `get_track`. [P0-A, R-INV-3]
- ⛔ **Движок рекомендаций** (поверх `alerts.py`) с инвариантами: нарушение → тип дороги → (КоАП на дорогах общего пользования / дисциплинарка СТ КАП на техдорогах) → частота → сумма штрафов → действие. [R-INV-1, R-INV-2, R-INV-6]
- ⛔ **Справочник статей/МРП в `config.py`** + `koap_revision_date`; **сверка КоАП на adilet** перед показом сумм. [R-INV-8] 🔴 юр.
- ⛔ **Shadow-период** — считать нарушения без публикации денежных карточек до валидации precision заказчиком.

## Фаза 3 — Операционные модули

- ⛔ **Контроль ТО** — наработка «от нуля» (T0) по моточасам/пробегу из фида + **T0-offset** при онбординге + заблаговременный алерт + **цикл подтверждения** (сброс). Спека [kb-11](knowledge-base/11-maintenance-to-tracking.md).
- 🟡 **Sensor Health — продакшен:** вывод диагнозов на портал, **плановый сбор baseline** (периодически фиксировать «здоровый» снимок), связать «Журнал»-диагноз с реактором (алёрт/тикет). Ядро готово.
- ⛔ **Реестр датчиков** (серийники из монтажных отчётов, валидация дублей/формата, хронология замен; `LLS_ID`=слот, не серийник).
- ⛔ **Push-доставка** — еженедельный дайджест ДЗО (топ-3 проблемных ТС + потери ₸ + действие); дашборд — для drill-down. [adoption]
- ⛔ **Рейтинг/лига ДЗО** — мотиватор adoption; фрейминг «ваш инструмент экономии», не «контроль сверху».

## Фаза 4 — Прод, масштаб, данные

- ⛔ **Идемпотентность фактов** — `fact_fuel`/`fact_events`/`violation` с natural key + UPSERT + `source_loaded_at` (8×/сутки + пересчёты Omnicomm). [P1-арх]
- ⛔ **Postgres до запуска факт-таблиц** (не «при росте»); порог перехода — перед violations/fuel.
- ⛔ **Версионирование `vehicle_org`** (`valid_from/valid_to`, `category`) — иначе KPI/лимит едут задним числом при переводе ТС.
- ⛔ **Health-check учётки** (права + доступность эндпоинтов; каталог уже отдаёт `permissions`) + SLA на копию КАП.
- ⛔ **ПДн/трансграничность** — подтвердить геолокацию копии (`online.omnicomm.ru` — РФ-домен) и каналов; правовое основание досье водителя. [R-INV-7] 🔴
- ⛔ **Досье водителя** (год рождения/стаж/ДОПОГ) — после правового основания; требует идентификации (iButton/RFID, в фиде есть `DRIVER_CODE`).
- ⛔ **Деплой** — cron+flock-вотчдог, прокси перед Streamlit, мониторинг (как в fleet-report); инфра в РК.
- ⛔ **Расширенный «Журнал»/ретрансляция** (EGTS/Wialon) — для сенсор-потока в реальном времени (опц., путь B [kb-10]).

## Приоритетная последовательность (рекомендация)

1. **Сейчас:** Sensor Health на портал (доверие) → seed геозон → детекция превышений (минимум к демо) — это и MVP-доказательство ценности, и подготовка Фазы 2.
2. **Параллельно (не код):** КАП Business Case + согласования (Дима/Антон/Данияр/юр).
3. **Затем:** MVP-доставка (Excel/RU-KK/мобильный) + безопасность (крипто-пароли) → онбординг первого ДЗО (вероятно Волковгеология) с тестом изоляции.
4. **После пилота:** движок рекомендаций (shadow→публикация), контроль ТО, идемпотентность фактов, Postgres.

> Главный технический риск (из аудита): детекция превышений на 2000 ТС — спроектирована на «триаж→ночной батч на сабсете», не «трек по всему парку». Главный продуктовый риск: MVP-упаковка без дифференциатора — закрывается срезом «скоростной режим в ₸» к демо. Главный правовой риск: КоАП на техдорогах — закрывается R-INV-1.

# Holding Architecture — целевой дизайн

Документ фиксирует **целевую архитектуру** платформы для холдинга и решения,
принятые при переходе от single-client отчёта (`omnicomm-fleet-report`) к
holding-scale системе. Это north star проекта; статус реализации — в `NOTES.md`.

## 1. Контекст заказчика

- Центральная учётная система холдинга (далее **КАП**).
- **23 ДЗО** первого уровня, данные дублируются из центральной системы.
- У части ДЗО (напр. «Уранэнерго», УМЗ) есть **под-ДЗО** (организации следующего уровня).
- У ДЗО бывают **подрядные организации**, чью технику тоже нужно выводить на дашборды.
- Сбор: скрипты из КАП → сервер в Астане → витрина. Обновление **8 раз в сутки**.
- Парк: ~1427 ТС (1035 активны за 7 дней на момент осмотра).

Текущая витрина — Power BI «Мониторинг скоростного режима-4» (4 страницы, только
скорость). Её ограничения и почему строим отдельную платформу — см. README.

## 2. Ключевое архитектурное решение: иерархия вместо «колонки на ДЗО»

Power BI-модель держит **отдельный столбец на каждое ДЗО** — это анти-паттерн:
добавление 24-го ДЗО, под-ДЗО или подрядчика = переделка модели. У холдинга на
деле **иерархия**, а не плоский набор колонок:

```
КАП (холдинг)
 └── ДЗО (×23): Уранэнерго, УМЗ, ...
      └── под-ДЗО: Уранэнерго → {ТФО, ШФО}; УМЗ → {УМЗ Курчатов}
           └── подрядная организация
```

Целевая модель — **звезда (star schema)**:

- `dim_org(org_id, parent_id, level, name, type)` — `level` ∈ {holding, dzo, sub_dzo, contractor}; `type` ∈ {own, contractor}.
- `dim_vehicle(vehicle_id, terminal_id, org_id, category, plate, passport_ref, valid_from, valid_to)`.
- `dim_geozone(geozone_id, name, polygon)`.
- `fact_events` / `fact_fuel` / `fact_motohours` — факты с `org_id` + `vehicle_id` + временем.

Добавить организацию = одна строка в `dim_org`. Любой разрез (по ДЗО, по уровню,
«своя техника vs подрядчик», роллап на КАП) = фильтр/group-by по измерению.
Это и снимает проблемы «маленьких витрин» и ручных выгрузок.

## 3. Тип ТС + лимиты скорости по геозонам

Omnicomm не умеет авто-определять тип ТС и динамически назначать лимит зоны по типу.
Решаем в слое обработки:

- `dim_vehicle.category` — жёсткий тег: Легковой / Грузовой / Буровая / Спецтехника
  (категорий закладываем больше двух). Ведём **с историей** (`valid_from/valid_to`) —
  машина может сменить роль, иначе пересчёт задним числом поедет.
- `geozone_limit(geozone_id, category, max_speed)` — N лимитов на зону по категории.
- Нарушение = джойн `факт × geozone_limit по (geozone, category)`.

Базис уже есть в `vehicle_types.py` (профили/классификация); добавляется таблица
`geozone_limit` и привязка к категории.

## 4. Чистка выбросов скорости (228–450 км/ч)

Пиковые скачки (трекер «телепортнулся» и вернулся) — классическая GPS-аномалия.
В Power BI их частично резали окном по длительности (10–90 с) — костыль: теряет
реальные короткие (<10 с) и устойчивые длинные (>90 с) нарушения.

Целевое решение — **валидация по физике движения на приёме**:

- берём две соседние GPS-точки, считаем `расстояние / Δt`;
- если подразумеваемая скорость/ускорение физически невозможны (телепорт-и-возврат) —
  **выбрасываем точку, а не всю запись**;
- аномалию **помечаем** (`sensor_error` / `severity=REVIEW`), **не удаляем молча** —
  для гос-холдинга чистка должна быть объяснимой (аудит-след: сколько и почему отброшено).

Базис — `validator.py` (уже помечает «требует проверки», не обвиняет; порог
`MAX_PLAUSIBLE_SPEED_KMH`). Добавляется межточечная физическая проверка трека.

## 5. Буровые и спецтехника: моточасы / топливо / акселерометр

Штатными отчётами Omnicomm не всегда достаётся, но сырьё есть в API (как события
у `omnicomm-fleet-report`/Романа). Деривация — в слое обработки, не из готовых отчётов:

- **моточасы** — из сигнала зажигания/оборотов (RPM>0 в геозоне стоянки = работа на месте);
- **топливо** — из CAN/датчика уровня бака (фильтрация сливов/заправок по скачкам уровня);
- **акселерометр** — журналы резких ускорений/торможений как отдельный тип события.

Базис — `loading.py` (работа спецтехники на месте, sensor→rpm→geozone→gps) +
`economics.py`/`norms.py` (топливо → ₸).

## 6. Подрядчики — доступ к данным (РЕШЕНО)

**Заказчик подтвердил (2026-06-18): вся техника — подрядчики и под-ДЗО — в ОДНОМ
фиде КАП.** Отдельных Omnicomm-логинов нет.

Следствия:
- Подрядчик/под-ДЗО — просто узлы в `dim_org` (`type=contractor` / `level=sub_dzo`).
- **Мульти-аккаунтный сбор НЕ нужен** — `auth.py` остаётся на одном логине, пул не строим.
- Вся сложность сводится к корректному разбору одного фида по `org_id` (маппинг
  ТС → организация), а не к интеграции N источников авторизации.

## 7. Доставка без лицензий

Power BI требует посемест­ную лицензию (≈8500 ₸/мес × на каждого) + ручную выгрузку
по воскресеньям. Поскольку данные автоматические (8×/сутки), раздаём отчёт прямо из
базы: web-дашборд / `.pptx` / `.xlsx` / `.html(+PDF)` на каждое ДЗО по расписанию.
Базис — `scheduler.py` + `mailer.py` (доставка без логина у получателя).

## 8. Контроль доступа (row-level) — конфиденциальность между ДЗО

**Жёсткое бизнес-правило заказчика:** руководитель холдинга видит общий дашборд
(весь КАП), а каждое ДЗО/подразделение видит **только свои** данные и **не видит
соседей**. В текущем Power BI это решалось либо посемест­ными логинами (дорого), либо
бесплатными ссылками Publish-to-web (тогда пропали и экспорт, и конфиденциальность —
анонимная ссылка видна кому угодно). Платформа делает контроль доступа **своим слоем.**

Модель: каждый пользователь привязан к узлу `dim_org` и видит **узел + всё поддерево** ниже:

- руководитель холдинга → корень КАП → весь холдинг;
- начальник ДЗО (напр. Уранэнерго) → Уранэнерго + ТФО + ШФО + **свои подрядчики**, но **не** УМЗ;
- под-ДЗО (ТФО) → только ТФО;
- **ДЗО видит технику своих подрядчиков** (подрядчик — дочерний узел этого ДЗО),
  но не видит подрядчиков чужого ДЗО; сам подрядчик (scoped-логин) видит только себя.

**Все подорганизации ДЗО — и под-ДЗО, и подрядчики — подвешиваются под свою ДЗО**
в `dim_org`. Поэтому «ДЗО видит свои подорганизации по иерархии» (и под-ДЗО, и
подрядчиков) получается из одной subtree-логики, без отдельных правил на каждый тип.

Изоляция «соседи не видят друг друга» получается **автоматически** из subtree-логики
(фильтр по поддереву `org_id` на каждом запросе/отчёте). Это и есть преимущество
иерархии над «колонкой на ДЗО». Доставка — персональный scoped-доступ к порталу или
персональный файл на ДЗО; **посемест­ных лицензий не требуется.** Так конфиденциальность,
экспорт и нулевая стоимость лицензий закрываются одним решением.

## 9. Источник данных (РЕШЕНО)

**Заказчик подтвердил (2026-06-18): всё тянем напрямую с API Omnicomm.** Не из БД и не
из файловых выгрузок. Один аккаунт Omnicomm = один фид КАП (и общий, и локальные
дашборды питались с него же). Базис — `api_client.py` + `auth.py` (auth/refresh/retry/
лимиты/коды ошибок уже есть). Подключаться к источнику текущего Power BI не нужно.

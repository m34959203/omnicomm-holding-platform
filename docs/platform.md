# Платформа и модули Omnicomm Fleet Report

Документ описывает актуальное состояние проекта: модули, конвейер, платформу
(мульти-клиент), паспорта/нормы/классификацию техники и ключевые особенности
интеграции с Omnicomm Online.

## Конвейер

```
источник → data_loader → validator → analytics → charts/svg_charts → report_builder → .pptx/.html(+PDF) → mailer
  ├─ Omnicomm API (consolidatedReport + track)        ▲                                   │
  ├─ Excel / CSV               norms/passports · vehicle_types · loading · price_history   │
  └─ клиент платформы (clients)  history · alerts · benchmark · fuel_price (календарь) ────┘
```

Всё после `data_loader` работает только с единой моделью (`models.py`) и не знает
про источник. Деньги — в тенге (₸).

## Omnicomm API — рабочий контракт (КРИТИЧНО)

Сверено с `developers.omnicomm.ru/api.yaml` и проверено на демо+боевом контурах.

- **Авторизация:** `POST /auth/login?jwt=1` с `{login,password,service}` → `{jwt,refresh}`;
  заголовок `Authorization: JWT <jwt>` (с пробелом). Для доступа к отчётам обычно
  нужен `service` (напр. `omnicomm`) и права на отчёты у учётки.
- **Дерево ТС:** `GET /ls/api/v2/tree/vehicle` — ТС вложены в `children[].objects[]`
  (рекурсивно), у объекта `uuid`, `name`, `terminal_id`, `terminal_type` (FAS/FTC).
- **Сводный отчёт:** `POST /ls/api/v1/reports/consolidatedReport` (НЕ GET!), тело
  `{"vehicleIds":[<terminal_id int>],"timeBegin":<unix>,"timeEnd":<unix>}`.
  Лимиты: **≤50 ТС и ≤31 дня** на запрос. Ответ — `items[]`, строка на **ТС × сутки**
  (`vehicleId`, `date`, блоки `mv`/`fuel`/`uniDataList`/`can`...). Имени нет — берём из дерева.
- **GPS-трек:** `GET /ls/api/v1/reports/track/{terminal_id}?timeBegin&timeEnd` →
  `[{date,latitude,longitude,speed,direction,satellitesCount}]`.
- **Геозоны:** `GET /api/service/geozones/geozones` (список), `POST /ls/api/v1/reports/geozones`
  (отчёт по визитам). У клиента геозоны не заведены.

### Единицы (частый источник ошибок ×10)

| Поле | Единица | Приведение |
|---|---|---|
| `mileage`, `maxSpeed` | км, км/ч | — |
| `worked`, `workedNoMovement`, `univInputOnTime` | секунды | /3600 = моточасы |
| `fuelConsumption`, `fuelConsumptionWOMovement`, `univInputOnConsumption`, `refuelling`, `draining` | **децилитры (дл)** | **/10 = литры** |
| `fuelCons100`, `fuelConsMH`, `fuelConsMHWOMovement` | **дл** | /10 |
| `deviation` | сл | /100 |
| `univInputHourConsumption` | спека пишет «л», по факту **дл** | считать из тоталов, не доверять полю |

Приведение к литрам/часам — в `data_loader._aggregate_consolidated` (Excel — уже в литрах).

## Модуль «Работа на погрузке» (`loading.py`)

Для мусоровозов/спецтехники: топливо списывается **и по моточасам** (работа
гидравлики на месте), не только по пробегу. Отделяет продуктивную погрузку от
непродуктивного простоя **даже без датчиков надстройки**. Источник определяется
**автоматически по достоверности**:

1. `sensor` — датчик доп.входа (`uniDataList`, `uniType` 0/1, `univInputOnTime>0`) — ФАКТ;
2. `sensor_zero` — датчик есть, не включался;
3. `rpm` — обороты **под нагрузкой** (`workedUnderLoadRPM`) при неподвижности по GPS = работает гидравлика (правило заказчика: >~1000 об/мин + стоит → погрузка). ОЦЕНКА «≈», ограничена временем стоянки;
4. `geozone` — визиты в заведённые геозоны-площадки клиента = обслуживание (журнал площадок);
5. `gps` — короткие остановки маршрута (обслуживание площадки) = погрузка, длинные стоянки = простой. ОЦЕНКА «≈»;
6. `none` — нет ни одного сигнала → цифру не выдаём.

**Правило оборотов:** Omnicomm отдаёт ВРЕМЯ в трёх полосах (idling/normal/workedUnderLoad),
не сырые об/мин → точный порог «1000» из API не извлечь; «1000» = ориентир полосы «под
нагрузкой» (`config.RPM_WORK_THRESHOLD`, настройка терминала).

**GPS-калибровка** (боевой трек): окно остановки-обслуживания `LOADING_STOP_MIN_SEC`..`MAX`
(60..900 с); `cluster_track_points(speed_eps=1.5)` — иначе джиттер скорости на стоянке
дробит её; `min_satellites=0` — на стоянке спутников часто 0. **GPS-карта точек погрузки
убрана** (по решению заказчика); при заведённых геозонах — журнал обслуживания площадок.

Дуальная норма: л/100 км в движении + **л/моточас работы стоя** (`fuelConsMHWOMovement`).

## Классификация техники (`vehicle_types.py`)

Реестр типов с профилями: основной параметр (л/100 км vs л/моточас), ждать ли
работу на месте. Тип из паспорта в приоритете, иначе авто-классификация по
названию и кинематике. **Шаблон типа управляет расчётом** (какую метрику показывать).
Шаблоны (`DEFAULT_PROFILES`) **редактируются из интерфейса** (режим «Шаблоны техники»)
— сохраняются в `data/vehicle_types.json` (переопределяют/дополняют дефолты).

## Аудит действий (`audit.py`)

Журнал в `data/audit/audit.jsonl`: кто/когда/что/по какому клиенту. Логируются
создание/правка клиента, подстановка цены, **сохранение паспортов/типов/норм**,
**правка шаблонов типов**, расписание, ручные и авто-прогоны. Просмотр — режим
«Журнал действий» (фильтр по клиенту).

## Паспорта и нормы (`norms.py`)

Хранятся по клиенту в `output/norms/<slug>.json` (enter-once). Паспорт: тип, марка,
модель, год, госномер, бак, двигатель, нормы, **коэффициент**. Модель списания:
ожидаемый расход = (пробег·л/100км + моточасы_стоя·л/моточас) × коэффициент →
отклонение факт−норма → **перерасход/экономия** в л и ₸. Коэффициенты (config
`NORM_COEFFICIENTS`): зима +10%, свалка +15%, город +10%, износ +5%. Без нормы
перерасход не утверждается (бизнес-инвариант).

## Доп. аналитика (по итогам совета директоров)

- **Авто-сигналы** (`alerts.py`): перерасход по ТС > порога ₸, доля холостого хода
  > порога, много «тёмных» ТС. Пороги — `config.ALERT_*`. Рассылка `alerts.send_alerts`.
- **Что если** (`analytics.build_whatif`): экономия при сокращении простоя на
  10/20/30% (`config.WHATIF_IDLE_CUTS`).
- **Скоринг ТС** (`analytics.build_scorecard`): балл проблемности (перерасход +
  холостой + превышения), худшие сверху. Драйвер-скоринг по водителям — нет данных.
- **Бенчмаркинг** (`benchmark.py`): сравнение клиента со средним по другим паркам
  (обезличенно, из снапшотов истории) — преимущество мультитенанта.
- **Сверка цены ГСМ** (`fuel_price.py`): см. раздел ниже.

## Рассылка по e-mail (`mailer.py`)

SMTP из ENV (`SMTP_HOST/PORT/USER/PASSWORD/FROM/TLS`). CLI: `--email` (отчёт),
`--alert-email` (сигналы), `--preset` для cron. Платформа: e-mail хранится у
клиента, чекбоксы «Отчёт»/«Сигналы», отправка при формировании; показывается
готовая cron-команда для авторассылки за период. Нет SMTP → предупреждение, не падаем.

## Роли и доступ (`auth.py`)

Платформа за **логином**. Две роли:
- **admin** — полный доступ: парк, нормы, клиенты, шаблоны, планировщик, пользователи;
- **manager** — только формирование и просмотр отчётов; конфигурацию не правит.

Пользователи — `data/users.json` (gitignored), пароли **PBKDF2-хеш** (не base64, в
отличие от sandbox-хранилища паролей Omnicomm-клиентов). Админ засевается при первом
запуске (пароль одноразово на экране входа или `ADMIN_PASSWORD`). Админ-экран
«Пользователи» создаёт/удаляет учётки. **Все действия логируются** с привязкой к
пользователю (`audit.set_actor` → поле `actor`); видно в «Журнале действий».

## Платформа (`clients.py` + `app.py`)

Мульти-клиент в песочнице:
- `data/clients/<slug>.json` (gitignored) — учётка Omnicomm (пароль обфусцирован
  base64, **не криптостойко** — песочница) + цена топлива + GPS-флаг.
- Режимы: «Сохранённый клиент» (выбрал → период → отчёт, учётка не вводится),
  «Новый клиент» (онбординг), «Файл (разово)». Кнопка «Сформировать» деактивируется
  на время генерации и показывает прогресс по стадиям + тайминг.
- Редактор паспортов/норм по клиенту: заполнил один раз → следующий отчёт сам
  показывает типы и перерасход/экономию. (только админ)

## Цена ГСМ: сверка и календарь (`fuel_price.py`, `price_history.py`)

**Сверка** — парсер цены дизеля у поставщика **Royal Petrol** (`royal-petrol.kz`,
Tilda zero-block через playwright; кэш 6 ч). Чек-поинт вида ДТ **лето/зима**; в
платформе авто-предупреждение при отклонении ручной цены > 15% и кнопка подстановки.

**Календарь цены ГСМ** (`price_history.py`) — история «с даты X цена Y»
(`data/fuel_prices.json`). При расчёте за период берётся цена **конкретного дня**,
итог — средневзвешенная по дням (учёт «взлётов/падений»: до 01.06 — 320 ₸, после —
340 ₸). Ручная цена клиента — фолбэк при пустом календаре. **Демон ежедневно
наполняет календарь** (`scheduler.snapshot_fuel_price`: раз в день парсит Royal Petrol
→ пишет цену дня; самогейт по дате + кулдаун 1 ч при сбое). Редактор — экспандер
«Календарь цены ГСМ» в платформе. Сбой парсера — снапшот тихо пропускается.

## Запуск

```bash
pip install -r requirements.txt
cp .env.example .env                 # учётка для CLI-режима API (опц.)
# CLI:
python -m omnicomm_report --source api --demo --preset last-month --fuel-price 320 --html --with-track
# Платформа (за логином; первый запуск засеет админа):
streamlit run app.py
# Демон авторассылки + ежедневный снапшот цены ГСМ:
python -m omnicomm_report.scheduler
```

## 1. Архитектура (слои)

```
Omnicomm Online (копия КАП projectkap, REST API, лимит ~180 req/мин на аккаунт)
        │  забор: дерево ТС · сводный отчёт (POST) · трек · геозоны · активность · state
        │  ⟂ глобальный rate-limiter ≤170/мин — платформа = ЕДИНСТВЕННЫЙ буфер к Omnicomm
        ▼
[1] Сбор/транспорт        api_client.py — JWT auth/refresh, батчинг ≤50 ТС/≤31 дня, ретраи, коды ошибок
        ▼
[2] Нормализация          data_loader.py — единый контракт VehicleMetrics; дл→л, сек→ч; чистка GPS (track_clean.py)
        ▼
[3] Иерархия организаций  org.py — dim_org (Холдинг→ДЗО→под-ДЗО→подрядчик), маппинг ТС→org_id, row-level доступ
        ▼
[4] Аналитика/роллапы     analytics.py · rollup.py — KPI по ТС и по поддереву; economics/norms/savings → ₸
        ▼
[5] Движки                geozone_limit · детекция превышений · рекомендации · контроль ТО · sensor health
        ▼
[6] Кэш-снапшот           api/cache.py — SQLite-снимок дашборда; чтения мгновенны, в Omnicomm НЕ ходят
        ▼
[7] Мост/витрина          FastAPI (api/) → Next.js (web/, static export) за реверс-прокси (deploy/holding_proxy.py)
        │                 executive-дашборд (scope-rail, 6 KPI, лента внимания, домен-табы) · карточка ТС
        │                 доставка: Excel · RU/KK · мобильный вид · выбор снимка/период
        ▼
Хранилище                 store.py — SQLite star schema (dim_org + vehicle_org) + кэш-снапшоты; Postgres при росте
                          объём: агрегаты ~1.5 ГБ/год · полные треки ~50–60 ГБ/год
```

Принцип: всё после `data_loader` работает только с `models.py` (источник абстрагирован). Константы/эндпоинты/коды — в `config.py`.

**Прод-стек (`omnicomm.technokod.kz`):** Python-движки → FastAPI-мост (`api/`) → Next.js фронт (`web/`) за реверс-прокси. Streamlit-портал (`holding_app.py`) выведен из критпути в legacy. Принцип буфера: чтение дашборда идёт из кэш-снапшота, а не из Omnicomm — поэтому **раздача ДЗО-учёток не множит нагрузку** на аккаунт Omnicomm.

## 2. Модель данных

**Существующее (star schema, `store.py`):**
- `dim_org(org_id, parent_id, name, type)` — иерархия организаций.
- `vehicle_org(terminal_id, org_id)` — привязка ТС к узлу (ключ — `terminal_id`, не uuid).
- `VehicleMetrics` — контракт телеметрии (mv/fuel/моточасы/трек/org_id).

**Новое (проектируется):**
- `geozone_limit(geozone_name, vehicle_category, max_speed, dzo)` — seed из СТ КАП ([09](knowledge-base/09-kap-geozone-speed-standard.md)); матрица-фолбэк (6 зон × 3 категории).
- `violation(terminal_id, ts, geozone, speed, limit, excess, range, koap_article, fine_kzt)` — факты превышений.
- `sensor_registry(serial, terminal_id, type, installed_from, installed_to, status)` — реестр датчиков с историей замен (ведётся вручную из монтажных отчётов).
- `to_plan(terminal_id, interval_mh, interval_km, remind_before)` · `to_state(terminal_id, mh_since_T0, km_since_T0, T0, status)` · `to_confirm(terminal_id, date, by, mh_at)` — контроль ТО.
- `fact_fuel` / `fact_events` — факты для трендов на масштабе (та же звезда).

## 3. Движки

### 3.1. `geozone_limit(геозона, тип_ТС)`
`лимит = min(лимит_ПДД(категория, дорога), лимит_геозоны(имя[, тип]), зонная_матрица(категория, зона))`.
Источник истины — именованные геозоны СТ КАП (~400, имя↔Omnicomm); координат в справочнике нет → геометрию брать из геозон Omnicomm по имени. Правило `min(внутр, ПДД)`, внутренний превышать нельзя. Перед загрузкой — нормализация/дедуп (Заречное с чужими именами, СТОП=0, дубли, OCR-топонимы — [09 §5](knowledge-base/09-kap-geozone-speed-standard.md)).

### 3.2. Детекция превышений по треку (с учётом масштаба и инвариантов)
**Двухступенчатый, не «трек по всему парку»** (трек = 1 GET/ТС, лимит ~150 req/мин → 2000 ТС/проход ≈ 14 мин; 8×/сутки нереализуемо):
1. **Дешёвый сигнал** — суточный `mv.maxSpeed` из сводного отчёта по всему парку → отобрать **ТС-кандидаты** (max_speed > мин. лимита зоны).
2. **Трек только по кандидатам**, отдельным **ночным батчем** (не в роллапе 8×/сутки). Добавить нарезку периода (`_period_windows`) в `get_track` (сейчас её нет → длинные периоды битые). Распараллелить под общий rate-budget.
3. Проверить, не отдаёт ли `get_geozones_report` факт превышения в визите (тогда трек нужен только для классификации).

Квалификация (инварианты [kb-05 §0](knowledge-base/05-recommendation-engine.md)): по **устойчивому сегменту** (≥N точек + фильтр ускорения), не по одной точке (R-INV-3); тип дороги обязателен (R-INV-1: техдорога → дисциплинарка СТ КАП, дорога общего пользования → КоАП); фильтр выбросов — ускорение + `satellitesCount≥4` (порог 200 недостаточен для зон 5–50). Чистка телепортов — `track_clean.py`.

### 3.3. Рекомендации «на букве закона»
Поверх `alerts.py`: сигнал → квалификация по [03](knowledge-base/03-legal-framework-kz.md) (ст. 592/571, МРП 2026=4325₸) → частота/повторность → сумма штрафов → вероятный ущерб (Nilsson + износ + простой) → действие. Лестница эскалации до дооснащения DSM/ADAS ([05](knowledge-base/05-recommendation-engine.md)). Справочник статей — в `config.py`.

### 3.4. Контроль ТО
Наработка «от нуля» (T0): аккумулировать моточасы (`mv.worked`) и/или пробег (`mv.mileage`/CAN-одометр) из фида → сравнить с интервалом → алерт → подтверждение ответственного сбрасывает цикл. Своё ведение (не Omnicomm-ТО). Детали — [11](knowledge-base/11-maintenance-to-tracking.md).

### 3.5. Sensor Health (реализовано — `sensor_health.py`)
- **Уровень 1 (терминал):** «светофор» ONLINE/STALE/OFFLINE по `activity/vehicles` (`dateID`) + `receive_data`. ✅
- **Грубая дата-валидация:** наличие возможностей (GPS/двигатель/ДУТ/CAN/доп.входы) по блокам `consolidatedReport` + детекция «потухло vs baseline» → исключение из KPI. ✅
- **Уровень 2 (сенсор по адресу + напряжение бортсети):** ✅ через **«Журнал»** (`POST /ls/api/v1/click/log`): `LLS_ID/LLS_CODE_PRESENT` (статус ДУТ по слоту) + `U_BOARD` (gate «питание есть + ДУТ молчит = сбой»). `get_journal()` + `journal_health()`/`diagnose_dut_failure()`. Звать **точечно** по ТС-подозреваемым (триаж уровнем 1). Серийник датчика — вручную (`LLS_ID` = слот, не серийник).
- **Оркестрация + baseline:** `select_suspects` (терминал жив + возможность пропала vs baseline) → `investigate` (точечный `journal_fetch`) → `diagnose_suspect`. Baseline здоровья (возможности + слоты ДУТ per-ТС) персистится в `store.sensor_baseline` (SQLite, upsert) — триаж сравнивает с историей, не с разовым снимком.

## 4. Доступ (row-level, две роли)

`auth.authenticate` → `{username, role, org_id}`. `role ∈ {admin(начальник), manager(ДЗО)}`. Видимость = `org.OrgTree.visible_scope` (admin=всё, manager=поддерево `org_id`). `filter_vehicles_for_viewer` — fail-closed. Перепроверка scope перед рендером в `holding_app.py` (defense-in-depth). Тонкого RBAC нет — все права = просмотр+экспорт в своём scope.

| Возможность | Эндпоинт | Статус |
|---|---|---|
| Авторизация JWT | `POST /auth/login?jwt=1` | ✅ |
| Дерево ТС (иерархия) | `GET /ls/api/v2/tree/vehicle` | ✅ (73 узла, ~1962 ТС) |
| Сводный отчёт (топливо/обороты/моточасы) | `POST /ls/api/v1/reports/consolidatedReport` | ✅ (блоки mv/fuel/fueladd/can/uniDataList) |
| Трек GPS | `GET /ls/api/v1/reports/track/{id}` | ✅ |
| Геозоны (визиты) | `POST /ls/api/v1/reports/geozones` | ✅ (~199 зон; источник детекции превышений сейчас) |
| Состояние ТС (**напряжение бортсети**) | `GET /ls/api/v1/vehicles/{id}/state` | ✅ найдено — `U_BOARD`/напряжение здесь, НЕ в consolidatedReport |
| Активность терминалов («светофор») | `GET /ls/api/v1/activity/vehicles` | ✅ (время последних данных) |
| **«Журнал» (сырьё по узлам)** | `POST /ls/api/v1/click/log` | ✅ доступен; 315 колонок/пакет с `*_PRESENT` (тяжёлый — точечно) |
| Доп. CAN/Modbus/польз. параметры | `POST /ls/api/v1/click/log/additional` | ✅ |
| По-датчиковый статус ДУТ + напряжение бортсети | `state` (`U_BOARD`) + «Журнал» (`LLS_*`, `IS_*_SUPPLY_*`) | ⚠️ напряжение/ур.1.5 ✅; глубже (темп. топлива/Журнал) — ограничено |
| **События вождения** | `GET /ls/api/v1/reports/events/{id}` | ⛔ на `projectkap` **пусты** (не провиженены — нужна учётка с событиями) |
| Каталог отчётов | `GET /ls/api/v1/reports/` | ⛔ закрыт правами (не мешает прямому `click/log`) |

**Защита аккаунта Omnicomm:** глобальный rate-limiter ≤170/мин на сервис-учётку (лимит Omnicomm ~180/мин) + TTL-кэш карточки ТС; чтения дашборда идут из кэш-снапшота (`api/cache.py`), не из Omnicomm.

**Единицы (гочи):** топливо в децилитрах → /10; время сек → /3600; `consolidatedReport` оборачивает данные в `{"consolidatedReport":{…}}` (разворот в `data_loader`); коды 5/7/9/10/11 не прерывают отчёт. См. [feedback Omnicomm API].

## 6. Ограничения и их следствия (актуально на 2026-06-23)

- **Детекция превышений — по визитам геозон, не по сырому треку.** Сейчас платформа определяет превышения через визиты геозон (`/reports/geozones`, 430 ТС в скоростном домене), а не через сырой трек с привязкой каждой точки к геозоне. Покоординатная детекция точка→геозона по треку — следующий кирпич (повысит точность квалификации сегмента).
- **События Omnicomm `/reports/events/{id}` на `projectkap` пусты** — не провиженены на этой учётке. Чтобы задействовать события вождения (акселерометр/SafeDrive), нужна учётка с включёнными событиями.
- **Сенсор-уровень: напряжение бортсети снято с ограничений**, найдено в `GET /vehicles/{id}/state` (питание есть + ДУТ молчит = сбой). Уровень глубже напряжения (температура топлива, по-датчиковый «Журнал») по-прежнему ограничен объёмом тяжёлого отчёта → точечно по подозреваемым.
- **Контроль ТО — наработка в пределах периода.** Сейчас наработка считается внутри окна синка; накопление между синками («от нуля» T0) — впереди.
- **Экономика COI завышается при аннуализации из коротких окон.** Перевод перерасхода в годовой ущерб из окна <7 суток раздувает COI → нужен гард (порог окна ≥7 сут) + явное разделение «измеримо» vs «оценка».
- **Серийники датчиков не в фиде** (`LLS_ID` = слот 1..6, не серийник) → реестр серийников вручную из монтажных отчётов, с валидацией дублей/формата; статус/живость датчика — из «Журнала»/`state`.
- **Видео (DMS/ADAS) не в Omnicomm** → отдельный контур; в платформе — потенциальный доп. источник.
- Профильные системы Казатомпрома недоступны.

## 7. Стек и деплой

- **Бэкенд:** Python 3.10+ · FastAPI + uvicorn (мост `api/`) · requests · pandas · openpyxl · matplotlib · python-pptx · SQLite (star schema + кэш-снапшоты `api/cache.py`). Streamlit (`holding_app.py`) — legacy, вне критпути.
- **Фронт:** Next.js 16 (static export → `web/out`), TypeScript, инлайн-SVG-графики, Яндекс.Карты, тема Omnicomm Online (цвета + Roboto, светлая/тёмная).
- **Прод:** `omnicomm.technokod.kz` — Python-движки → FastAPI (`api/`) → Next.js (`web/`) за реверс-прокси (`deploy/holding_proxy.py`); инфраструктура в РК. Деплой — см. [DEPLOY-holding.md](DEPLOY-holding.md).
- **Безопасность:** креды из ENV (`LOGIN/PASSWORD/SERVICE/OMNICOMM_BASE_URL`), `.env`/`data/`/`output/*` в `.gitignore`; пароли клиентов — шифрование в проде.

## 8. Риски

> Полный приоритизированный разбор рисков (P0/P1/P2) и **красные линии (R-INV-1…8)** — в [REVIEW-2026-06-22](REVIEW-2026-06-22.md) и [kb-05 §0](knowledge-base/05-recommendation-engine.md). Ключевые:

| Риск | Митигация |
|---|---|
| **Обвинение водителя по битым данным** (ложный лимит + GPS-выброс + техдорога + фиктивная сумма) | инварианты R-INV-1…5; shadow-период; квалификация по сегменту |
| **Подмена дисциплинарки СТ КАП штрафом КоАП** на техдорогах | R-INV-1: тип дороги — обязательный вход квалификации |
| Ложные нарушения из GPS-выбросов | R-INV-3: сегмент + фильтр ускорения (порог 200 недостаточен для зон 5–50) |
| Деньги на данных без сенсор-валидации ДУТ | R-INV-4: confidence-флаг; терминальный Sensor Health в MVP |
| Аномалии справочника геозон (дубли 40/80, Заречное, опечатки) | R-INV-5: verified seed с ревью заказчика, не авто-парс имени |
| **Идемпотентность фактов** (8×/сутки + пересчёт Omnicomm) | natural key `(terminal_id,date,…)` + UPSERT + `source_loaded_at`; «последний прогон выигрывает» |
| Контроль ТО «от нуля» дрейфует/двоится | наработка = `SUM(факт)` от T0 (не инкремент); T0-offset при онбординге |
| Зависимость от одной service-учётки (права режут молча) | health-check учётки (права+эндпоинты); SLA на копию КАП |
| ПДн/трансграничность (`online.omnicomm.ru` — РФ-домен) | подтвердить геолокацию копии и каналов; основание ПДн досье (R-INV-7) |
| Нет доступа к «Журналу» → нет сенсор-детекции | терминальный уровень сейчас; запрос расширенного доступа |
| Рост до 2000 ТС × история | Postgres до запуска факт-таблиц (не «при росте»); версионировать `vehicle_org` |

## 9. План реализации (следующие кирпичи)

1. Seed-датасет геозон из СТ КАП (имя→ДЗО→лимит→[тип]) с чисткой аномалий.
2. `geozone_limit` + детекция превышений по треку.
3. Движок рекомендаций на букве закона (config-справочник статей/МРП).
4. MVP-приоритеты: Excel-кнопка в портале, i18n RU/KK, мобильный вид, настройка частоты обновления.
5. Терминальный Sensor Health (`activity/vehicles`) + грубая дата-валидация.
6. Контроль ТО (наработка от нуля + цикл подтверждения).

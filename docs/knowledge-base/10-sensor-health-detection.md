# Детекция сбоев датчиков и качество данных (Sensor Health)

> Способность платформы: **автоматически выявлять неисправные датчики** (ДУТ, GPS, CAN и др.) по фиду Omnicomm — то, что сейчас аналитик ловит вручную по «обрыву» графика.
> Основано на уточнениях Антона (техспециалист интегратора, аудио 22.06) + разбор мануала [06](06-omnicomm-online-capabilities.md).

## 1. Проблема (со слов Антона)
- **Прямой индикации сбоя ДУТ в Omnicomm Online нет.** Аналитик/диспетчер замечает сбой **по графику топлива** — он просто перестаёт строиться. С графика снять это программно нельзя.
- Есть **«светофор» над списком техники** (🟢🟡🔴 кубики) — но это давность данных **терминала целиком**, а не статус конкретного датчика.
- **Источник для авто-детекции — отчёт «Журнал»**: в него приходят данные от ВСЕХ узлов (ДУТ, GPS-антенна, CAN…) с привязкой к **сетевому адресу**. Логика: «на технике ДУТ с такого-то адреса; данные по адресу перестали приходить → детектировать сбой ДУТ».

## 2. Почему это критично для платформы (наша рамка)
**Сбой датчика молча портит аналитику.** Если ДУТ сломан — топливные KPI по этому ТС мусорные (нет расхода/сливов/заправок); если GPS — нет трека и превышений; если CAN — нет моточасов/оборотов. Значит **детекция здоровья датчиков = вентиль качества данных ПЕРЕД аналитикой**: битые ТС надо помечать и **исключать из KPI/рейтингов**, иначе платформа врёт. Это и наш value-add (Omnicomm даёт только терминальный светофор), и сервисный сигнал (повод на ТО/проверку — ср. лестница эскалации [05](05-recommendation-engine.md)).

## 3. Что Omnicomm уже даёт (примитивы для детекции)
| Примитив | Уровень | Где |
|---|---|---|
| Флаг `receive_data` в дереве ТС | терминал | `GET /tree/vehicle` (видели на смоуке) |
| «Светофор» / фильтры активности («не поступали более», «данные отсутствуют») | терминал | дерево объектов, UI |
| Статусы LLS: **Готов / Нет / Ошибка / Не готов** | датчик (ДУТ) | отчёт «Журнал», по LLS1–6 |
| Событие **«Сбой LLS» / «Окончание сбоя LLS»** | датчик (ДУТ) | отчёты «Нарушения» / «События» |
| GPS: «нет валидных данных» (<4 спутников, 60 с), **детект глушения GPS/GSM** | датчик (GPS) | «Журнал» |
| CAN: OBDII **Check Engine**, наличие параметров шины | датчик (CAN) | «Журнал» |
| Сырьё по всем узлам с сетевым адресом | датчик (любой) | **отчёт «Журнал»** |

→ Для ДУТ Omnicomm уже умеет «Сбой LLS» как событие; наша задача — **поднять его автоматически + распространить на ВСЕ датчики по адресам + развязать с «терминал офлайн»**.

## 4. Модель детекции (двухуровневая)

```
Уровень 1 — ТЕРМИНАЛ (светофор):
    есть ли вообще данные от ТС? (receive_data / давность последней точки)
    🔴 нет данных дольше T_term  →  «терминал офлайн» (связь/питание/терминал), НЕ сбой датчика

Уровень 2 — ДАТЧИК (по сетевому адресу, из «Журнала»):
    для ТС известен инвентарь датчиков {адрес → тип}
    по адресу X нет данных дольше T_sensor,
        ПРИ ЭТОМ терминал онлайн (другие узлы шлют)  →  СБОЙ датчика X
    + прямые статусы (LLS=Ошибка/Нет, событие «Сбой LLS», GPS-глушение, CAN Check Engine)

Уровень 1.5 — ПИТАНИЕ (ключевое уточнение Дмитрия, 22.06):
    напряжение бортсети — лучший прокси «терминал жив и запитан».
    БЕЗ напряжения бортсети данных нет вообще → это не сбой датчика.
    Правило: напряжение_бортсети > порог  И  показаний ДУТ нет  →  СБОЙ ДУТ
    (питание есть, пакет идёт, а конкретный узел молчит = умер именно датчик)
```

**Шаги реализации:**
1. **Инвентарь датчиков на ТС** — `{сетевой_адрес → тип (ДУТ/GPS/CAN/УВ…)}`. Источник: конфигурация терминала (Omnicomm Configurator) **или** вывести эмпирически — какие узлы слали данные за «здоровый» baseline-период. **Развязка по питанию:** использовать напряжение бортсети как gate (см. Уровень 1.5) — без него «нет данных ДУТ» неотличимо от обесточенного ТС.
2. **Baseline нормальной работы** — по каким адресам и с какой частотой идут данные у исправного ТС.
3. **Детект сбоя датчика** — данные по узлу отсутствуют > `T_sensor`, а терминал онлайн → флаг «сбой <тип> на ТС». Развязка с уровнем 1 обязательна (иначе офлайн-терминал даст ложные сбои всех датчиков).
4. **Классификация влияния на аналитику** — сбой ДУТ → топливные KPI ТС невалидны; GPS → нет трека/превышений; CAN → нет моточасов. Пометить и **исключить из рейтингов/роллапов**.
5. **Вывод** — дашборд **«Здоровье датчиков»** на ДЗО: список ТС с битыми датчиками (тип, давность, влияние) + рекомендация «проверка/ТО датчика».

## 4а. Реестр датчиков с серийниками и хронологией замен (запрос Дмитрия)

Желаемый результат (дословно): **индикация «работает датчик или нет» + его серийный номер + возможность заменить датчик с хронологией** — «до такого-то числа стоял датчик A, после — датчик B, на какой технике он умер».

**Развилка по идентификатору датчика:**
- **Идеальный путь (проверить!):** терминал отправляет на сервер пакет (вероятно JSON) — если в нём есть **ID/статус конкретного датчика**, привязка серийников вручную НЕ нужна — берём ID + статус прямо из контроллера. Дмитрий: «насколько помню, что-то подобное должно быть».
- **Реальность (Антон):** Omnicomm, скорее всего, **серийный номер датчика на сервер не отдаёт** (запрашивал IMEI модемов — «не ведём, не фиксируем»); глубокого администрирования у интегратора нет (готовое решение с правами доступа, глубокие возможности — у вендора в Москве). «Адреса» в Журнале — это **сетевые адреса узлов**, не серийники.
- **Фолбэк-реестр (если ID нет в фиде):** вести справочник `датчик(серийник) → ТС → период (с/по) → статус`. Источник серийников — **отчёт монтажа** (монтажники предоставляют, какое оборудование/серийники установлены на технику).

**Защита от человеческого фактора (важно):** серийники монтажники пишут руками → опечатки. В Omnicomm терминал нельзя завести с дублирующим ID (система ругается), а **датчик — это «комментарий», можно вписать одинаковый в каждую технику** (ни на что не влияет, контроля нет). Поэтому наш реестр должен: валидировать формат серийника, ловить дубли (один серийник на двух активных ТС = ошибка), вести историю замен как версионируемые записи, по возможности — сверять с фактом «датчик жив по фиду».

**Вопрос даты:** «дата установки = первое появление или последнее?» — определить: дата установки = первое появление узла в фиде после монтажа; дата «смерти» = последнее появление перед длительным молчанием (при живом питании).

## 5б. Доступ к «Журналу» — РЕШЕНО (был неверный диагноз)

> Каталог отчётов `GET /reports/` закрыт правами — но это НЕ блокирует прямой вызов `POST /ls/api/v1/click/log`. Путь A (запрашивать права) для базовой детекции **не нужен** — доступ уже есть. Запрос расширенных прав/«Руководства по администрированию» остаётся полезным для полноты, но не блокирует.

**Путь A — выдать права/доступ (основной, дешёвый если согласуют).**
Запросить у дилера/Omnicomm (учётку администрирует интегратор; глубокая админка — у вендора): (1) «Руководство по администрированию» с REST-спекой; (2) грант на отчёты/«Журнал»/messages для учётки `projectkap` **или** отдельный API-ключ/роль с доступом к сырью. Разблокирует и каталог, и эндпоинт «Журнала». Владелец: Антон → Omnicomm.

**Путь B — ретрансляция сырого потока (мощный, обходит report-API целиком).**
Терминалы Omnicomm умеют **ретранслировать телеметрию на сторонний сервер** стандартными протоколами (EGTS / Wialon IPS / NDTP). Попросить интегратора настроить ретрансляцию парка КАП (или сабсета) на НАШ приёмник → получаем сырой пакет со всеми узлами (ДУТ по адресу, напряжение бортсети, статусы) в реальном времени, независимо от прав отчётов. Это архитектурно правильный ответ на «нужен сырой фид» (директор уже намекал: «получить доступ к серверам в таком же формате», аудио 18.08). Цена: нужен приёмник-сервер + парсинг протокола + согласие интегратора. Владелец: Антон + мы.

**Путь C — Sensor Health v1 на доступных данных (✅ РЕАЛИЗОВАН 22.06).**
Без «Журнала» уже реализуемо: терминальный светофор (`activity/vehicles` + `receive_data`) + по-возможностная детекция «данные потухли vs свой baseline» из блоков `consolidatedReport` (пустой `fuel` у ТС, где ДУТ был → вероятный сбой ДУТ; `can`/`uniDataList` null → нет CAN/доп.входов) + GPS-health из трека (нет точек / мало спутников). Покрывает главный кейс (мёртвый ДУТ, офлайн-терминал) на суточной гранулярности. Не закрывает: точный адрес узла и развязку по напряжению бортсети (это даёт A/B).

## 6. Типы подключения к CAN-шине (со слов Антона — для каталога)
Подключение к CAN зависит от ТС и рекомендаций вендора (влияет на доступность/надёжность CAN-данных):
- **«Крокодилы»** (бесконтактный считыватель, накладка на провода) — рекомендация вендора, без врезки в шину.
- **CAN-LOG** — бесконтактный считыватель (Mechatronics), подключение без разрыва шины.
- **Прямое подключение** к CAN-шине.
→ При диагностике «нет CAN-данных» учитывать способ подключения (бесконтактный считыватель может терять связь). Дополняет [01 §2.2 / §3.1](01-sensors-and-systems-catalog.md).

## 7. Связь с другими модулями
- Это **вентиль качества перед** роллапами [rollup] и рекомендациями [05] — битые ТС не должны попадать в KPI/рейтинги.
- Сбой датчика = ещё один тип события для движка алёртинга по модели ЦМ [07] (приоритет, SLA, эскалация в техслужбу — как «заявка на проверку камеры» в регламенте).

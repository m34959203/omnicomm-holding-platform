# Операционная модель Центра мониторинга (регламент оператора ЦМ IT-Transport)

> Разбор `download/Регламент обработки событий для оператора ЦМ IT-TRANSPORT.docx`.
> **Scope-пометка:** фактически документ — регламент **видеомониторинга состояния водителя** комплексом «Антисон» (камера DMS в кабине), НЕ трекинговой телематики Omnicomm. Но операционная модель ЦМ (очередь → приоритет → SLA → модерация → эскалация → журнал) — **готовый каркас движка алёртинга/рекомендаций** для нашей платформы.
> Источник извлечён 2026-06-22.

## 1. Таксономия событий (11 типов карточек, формирует устройство)
Засыпание · Усталость/зевание · Отвлечение (взгляд в сторону) · Курение · Телефон · Ремень не пристёгнут · Водитель отсутствует · Камера закрыта · **Выдача топлива** · Предупреждение о столкновении с пешеходом · Предупреждение о лобовом столкновении.
(Последние три — только в списке, в регламенте не раскрыты. Раскрыты 10 событий состояния водителя.)
Правило: *оператор валидирует, правильно ли устройство классифицировало событие («оценку производить исходя из наименования»).*

## 2. Механизм обработки
Устройство формирует карточку с видеороликом → оператор просматривает → подтверждает («корректная тревога») / отклоняет («ложная тревога»). Параметр **X** = порог длительности (закрытие глаз/отвлечение, сек) — задаётся договором. Сквозной критерий отклонения: **«ТС не движется»** и **«событие пришло с задержкой»**.

**Ключевой алгоритмический паттерн (4.2):** если за **2 часа** зафиксировано **3 события «Закрытие глаз» на скорости** → ТС помечается «**повышенный риск засыпания**». Работает и на 0 км/ч.

## 3. SLA / KPI и приоритеты (Приложение 2)
| Событие | Приоритет | SLA, сек | KPI распред., сек | KPI просмотра, сек |
|---|---|---|---|---|
| Засыпание | **1** | 120 | 90 | 22 |
| Многократные засыпания | **2** | 160 | 130 | 52 |
| Взгляд в сторону | 3 | 240 | 180 | 22 |
| Телефон | 3 | 240 | 180 | 12 |
| Курение | 4 | 300 | 240 | 12 |
| Ремень | 4 | 300 | 240 | 12 |
| Усталость (зевание) | 4 | 300 | 240 | 9 |
| Нет в кадре | 5 | 600 | 240 | 9 |
| Заслон камеры | 5 | 600 | 240 | 9 |
| Заслон при включении | 5 | 600 | 240 | 9 |

Логика: критичнее (засыпание) → жёстче SLA (120 с). «Допустимое время ожидания по клиентам» — таблица в документе пустая (задаётся индивидуально).

## 4. Роли и процесс
- **ЦМ** — центр мониторинга. **Оператор** — валидация (модерация) событий.
- **Менеджер диспетчерской службы** — при тревоге (засыпание/ДТП) звонит заказчику.
- **Диспетчер** — на стороне клиента. **Сопровождающий сотрудник** — эскалация, если не дозвонились. **Ситуационный центр** — получает инфо (напр. «ТС на буксире»).
- Вход: `https://it-transport.online:9965/vss/?#` → «Доказательства» → «Тревожные события» → выбор компании, фильтры (дата/тип/статус) → «Получить итог».
- Карточка: клиент, оргструктура (→ номер диспетчерской), дата/время, **скорость по GPS**, тип, видеоролик, борт.№, кнопки «Обработка/Удалить/Скачать».

## 5. Эскалация по задержке видео (критично для автоматизации)
| Задержка | Действие |
|---|---|
| >15 мин и ≤12 ч | модерировать + **звонить с доп.комментарием** о нестабильной связи |
| >12 ч и ≤72 ч | модерировать, но **звонок НЕ совершать** |
| любая | задержка = основание для отклонения большинства событий |
Прочее: не дозвонились → сопровождающему сотруднику · идут испытания → по рекомендациям · ТС на буксире → подтверждать, но не звонить.

## 6. Журналирование
Статус события (корректная/ложная/в ожидании) · сводный «итог» по компании/дате/типу · скачивание/удаление карточки · **заявки** в техслужбу (проверка камеры / юстировка) · флаг «повышенный риск засыпания» · метрики SLA/KPI.

## 7. Что автоматизируем в нашей платформе
| Пункт регламента | Автоматизация софтом |
|---|---|
| Паттерн «3 закрытия глаз / 2 ч» | алгоритм-правило: агрегация по ТС в скользящем окне → авто-флаг «повышенный риск» |
| Приоритеты + SLA-таймеры | очередь с приоритетами 1–5, обратный отсчёт (120…600 с), авто-эскалация, дашборд «горящих» |
| Градация по задержке (15м/12ч/72ч) | авто `now − event_time` → подсказка сценария (звонить/молча/отклонить) |
| Фильтр «ТС не движется» | кросс-проверка GPS-скорости → авто-пометка «вероятно ложное» |
| Эскалационная цепочка | авто-маршрутизация оператор→диспетчерская→сопровождающий→ситуационный центр + лог дозвонов |
| Скрипты обзвона | автоподстановка борт.№/времени в шаблон, авто-уведомление при подтверждении |
| Заявки на камеру | авто-тикет в техслужбу |
| Метрики SLA/KPI | авто-замер трёх таймингов, отчётность по операторам/клиентам |

**Стыковка с нашим scope:** рамка ЦМ (очередь→приоритет→SLA→эскалация→журнал статусов→авто-уведомление) переносится 1:1 на «классические» телематические события (превышение скорости, слив, отклонение от маршрута, выход из геозоны, простой, тревожная кнопка, потеря связи).

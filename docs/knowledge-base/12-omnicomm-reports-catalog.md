# Каталог отчётов Omnicomm Online (UI) и доступность через API

> Полный аудит того, что система умеет формировать в интерфейсе, с привязкой к
> задачам платформы и честной колонкой «доступно в UI / через REST API».
> Источник: ручной аудит UI на контуре `projectkap` (≈1998 ТС, ~30 юрлиц в дереве),
> меню **«Добавить отчёт»** — 4 группы, ~39 шаблонов. Дата: 2026-06-22.
> Связано: [06](06-omnicomm-online-capabilities.md) (возможности), [10](10-sensor-health-detection.md) (Sensor Health), [11](11-maintenance-to-tracking.md) (ТО).

## 0. Зачем этот файл

Снимает спор «что у Omnicomm есть» от «что доступно нам программно». **Многое есть в UI,
но закрыто по REST** на текущей учётке: отчёты `/reports/*` и «Журнал» отдают 404/нет прав;
`consolidatedReport` НЕ отдаёт напряжение бортсети, серийники/адреса узлов, `uniDataList`.
Поэтому ключевые для Sensor Health источники (Журнал, Напряжение бортсети) надо подтверждать
**отдельно** — выгрузкой из UI или расширением прав/другим методом доступа.

**Легенда колонки «API»:**
`✅` доступно программно сейчас (проверенный эндпоинт) ·
`⛔` есть в UI, но по REST закрыто/404/нет прав ·
`⚠️` не подтверждено — нужна проверка (нет в `consolidatedReport`, возможна выгрузка из UI) ·
`—` не применимо к нашему пайплайну.

## 4. Рабочие столы (9)

Комбо «Трек + N»: **Движение · Нарушения · Объём топлива · Объём топлива + Заправки/сливы ·
Работа доп. оборудования · Скорость · Скорость + Объём топлива + Заправки/сливы · События · Статистика**.

Это композиции UI поверх уже перечисленных источников (свой формат «рабочего стола»). **Для платформы прямой
ценности нет** — мы строим собственные дашборды; доступность через API определяется доступностью входящих
отчётов (трек ✅ + соответствующий слой). Полезны как референс компоновки «карта + телеметрия».

## 5. Сводка для Sensor Health (главный вывод)

Из [10](10-sensor-health-detection.md) нужны 4 источника — статус доступности:

| Нужно для Sensor Health | Шаблон UI | Уровень детекции | API сейчас | Что разблокирует |
|---|---|---|---|---|
| Сырьё по узлам + адреса | **Журнал** | ур.2 (сбой конкретного датчика) | ⛔ | права на `/reports/` ИЛИ выгрузка из UI |
| Питание терминала | **Напряжение бортовой сети** | ур.1.5 (gate «сбой ДУТ vs обесточен») | ⚠️ | подтвердить: фид/выгрузка/расширенный отчёт |
| События ДУТ | **Нарушения** / **События** («Сбой LLS») | ур.2 (прямой статус) | ⛔ | права на `/reports/` |
| Терминальный светофор | дерево ТС / `activity/vehicles` | ур.1 (терминал онлайн) | ✅ | — реализовано (R7.1) |

**Вывод:** уровень 1 (терминальный светофор) у нас уже работает в проде. Уровни 1.5/2 (питание + сбой
конкретного датчика) **остаются заблокированы доступом** — всё нужное есть в UI, но не подтверждено по API.
Следующий шаг — точечная разведка двух отчётов (Напряжение бортсети, Журнал): какие именно поля отдают и
есть ли экспорт в Excel. До этого Sensor Health честно держим на терминальном уровне ([10](10-sensor-health-detection.md), [SOLUTION §Ограничения](../SOLUTION.md)).

## 6. Открытые проверки (drill-down в UI)

1. **Напряжение бортовой сети** по одному ТС → поля, единицы, гранулярность (по пакетам / агрегат), экспорт Excel.
2. **Журнал** по одному ТС → колонки, наличие сетевого адреса узла и типа датчика, статусы LLS, экспорт Excel.
3. **Работа дополнительного оборудования** → подтвердить, отдаётся ли PTO/гидравлика отдельным отчётом (в обход отсутствия `uniDataList` в сводном).

Результаты этих проверок дописываются сюда (колонка API) и в [10](10-sensor-health-detection.md).

# Техническое предложение IT-Transport (ИВТ) — разбор и позиционирование

> Разбор `download/Техническое_предложение_ИВТ_IT-Transport_черновик.pdf` (ред. 2.0 от 12.06.2026, ООО «IT-Transport», www.it-transport.ru).
> Источник извлечён 2026-06-22.

## TL;DR
Это предложение **не про аналитику и не про Omnicomm** — это **«железо»: интеллектуальный видеомониторинг (ИВТ)** — MDVR + DMS (контроль водителя) + ADAS (дорога) + монтаж + базовый веб-портал событий. IT-Transport продаёт **оборудование + базовую FMS-платформу видеособытий**, не холдинговую BI-аналитику. Наш слой (holding-платформа поверх Omnicomm с ИИ-рекомендациями и кросс-ДЗО дашбордами) **встаёт сверху и рядом** — дублирования мало, главный их пробел закрываем мы.

## 1. Кто и для кого
- **Исполнитель:** ООО «АйТи-Транспорт» (российская компания/интегратор).
- **Заказчик:** обезличен («Компания заказчика ___»), но жёсткая привязка к **законодательству РК** + требование **исключить трансграничную передачу ПДн** (инфраструктура в РК). Согласуется с КАП/ДЗО.
- В глоссарии FMS = «система управления автопарком / **сторонняя** платформа» → они допускают чужую верхнюю систему. **Это наша дверь.**

## 2. Состав (борткомплект на 1 ТС)
MDVR **ITR-5SD-01AI** (5-канальный, Linux, 4×AHD 1080p + 1 IP, GPS+ГЛОНАСС, 6-осевой G-сенсор, 2×SD до 1 ТБ, 4G/Wi-Fi, **-20…+70 °C, IP54**) · камера **DMS** (контроль водителя) · фронт-камера **ADAS** · монитор **ITR VMV 002** 10,1" · доп. AHD **ITR AHD 001** (IP67, -45…+60 °C) · кабели/SIM/SD.
Услуги: поставка → монтаж → пусконаладка → калибровка DMS/ADAS/G-сенсора → активация на сервере → сервис.
> ⚠️ Темп. риск: MDVR -20…+70 «по даташиту», расширенное исполнение — по отдельному подтверждению. Для зимы РК (-40) — риск на их стороне.

## 3. Архитектура (4 уровня)
Борт (MDVR+DMS+ADAS, edge-детекция на борту) → канал (3G/4G, сервер **IT-TRANS/FMS**) → сервер (платформа, база событий, видеоархив) → пользователь (веб-портал).
Поток события: борт детектит → предупреждение водителю → локальная запись видео+атрибуты → отправка на сервер (буфер при потере связи) → диспетчер валидирует.
Атрибуты события: видео ≥5 сек, дата/время, тип, борт.№, **GPS-координаты + скорость**, статус.

## 4. Функционал
- **DMS (кабина):** отвлечение, усталость/зевание, засыпание, перекрытие камеры, ремень, телефон, курение, питьё, очки/маска.
- **ADAS + G-сенсор:** выход из полосы, пешеход, опасное сближение, резкое ускорение/торможение, **события геозон (въезд/выезд/скорость)**.
- **Платформа:** Дашборд, Просмотр событий, Онлайн-видео, Геозоны, Центр событий, **Отчёты** (GPS, тревоги, DMS/ADAS, скорость, онлайн/офлайн, пробег, топливо, активность водителя, температура), удалённая прошивка/калибровка, интерком.
> «Топливо» у них — лишь **строка в меню отчётов**, не топливная аналитика (нет уровня Omnicomm: расход/сливы/ДУТ).

## 5. Скорость / нарушения / геозоны
- Скорость — атрибут события (GPS); есть отчёт и правило «скорость в геозоне». **Полноценного контроля скоростного режима по сегментам/лимитам дорог НЕТ** — только превышение в геозоне.
- Нарушения = «события» с фото/видео + **обязательная ручная валидация диспетчером**. Достоверность детекции ≥90% помечена «подлежит проверке».
- **Агрегации нарушений по водителям/ДЗО с трендами/рейтингами как продукта — НЕТ.** Это наш слой.

## 6. Коммерция
- **Цифр НЕТ** — ни цен, ни сроков, ни количества ТС («по итогам обследования»).
- Модель: оборудование (CAPEX) + монтаж/ПНР + сервис/мониторинг (OPEX, подписка подразумевается). Гарантия ≥12 мес.
- Этапы: обследование → согласование → монтаж → ПНР → активация → приёмка (чек-лист 10 п.) → акт → сервис.
- Открытые вопросы: кол-во ТС/камер, период хранения видео, серверная инфра + ПДн, регламент событий.

## 7. ИИ / аналитика
- Дашборд — **операционно-диспетчерский** (статусы/события), не управленческая BI.
- «ИИ» = **edge-детекция на борту** (DMS/ADAS компьютерное зрение), НЕ аналитический ИИ платформы.
- **ИИ-рекомендаций, скоринга водителей, прогнозов, предиктивной аналитики — НЕТ.** Система реактивная.
- **Кросс-ДЗО / холдингового среза НЕТ** (иерархия максимум «Компания → Парк → ТС»).

## 8. Где мы добавляем ценность
**Их сильные стороны:** полный железный стек DMS+ADAS+MDVR под ключ (мы это не делаем); edge-детекция + видеодоказательства; учтена РК-специфика ПДн.
**Их пробелы = наши возможности:**
1. Нет аналитического/ИИ-слоя (рекомендации, скоринг, прогнозы, паттерны).
2. **Нет холдингового/кросс-ДЗО уровня** — критично для КАП (23 ДЗО).
3. Черновик без цифр.
4. Платформа — generic FMS производителя, готовы к интеграции с чужой верхней системой.
5. «Топливо» — строка меню, не продукт.
6. Темп. риск железа для зимы РК.
7. Скорость ограничена геозонами, нет дорожно-сегментного контроля.

**Вывод:** IT-Transport = «глаза и нервы» автопарка (видеоконтроль водителя/дороги, отдельный контур, в Omnicomm не передаёт — см. [holding-product-requirements §3](../holding-product-requirements.md)). Наша платформа = «мозг и память холдинга» (аналитика, ИИ, кросс-ДЗО). Пересечение поверхностное, взаимодополнение сильное. **Их события (GPS+тип+видео) — ещё один источник данных для нас**, наравне с Omnicomm; они сами называют FMS «сторонней платформой».

# Паритет отчётов Omnicomm ↔ наша платформа (бэклог)

> Цель (Дмитрий, 25.06): **каждая форма отчёта Omnicomm должна иметь аналог у нас.**
> Источник форм — [13-omnicomm-reports-audit.md](13-omnicomm-reports-audit.md) (аудит Cowork,
> 16 форм + «Объём топлива»). Статус сверен с кодом `src/omnicomm_report/` + `web/` на 25.06.
> Приоритет = популярность (полки Omnicomm «Популярные») × ценность для КАП × реализуемость
> на нашем фиде (часть упирается в доступ REST / железо — см. [12](12-omnicomm-reports-catalog.md)).

## Сделано (25.06, data/API-слой)

Построены отчётные формы из данных на руках (`reports.py`, секции снапшота + эндпоинты,
тесты `test_reports.py`). UI-компоненты — следующий слой:
- **Посещение геозон** → `build_geozone_visits` + `GET /api/geozone-visits` (таблица визитов + сводка из `fact_visit`).
- **Сводный / Работа группы** (посуточный итог по ТС, все метрики) → `build_fleet_table` + `GET /api/fleet-table`.
- **Нарушения** → `build_violations` + `GET /api/violations` (геозон-превышения со статьёй КоАП + агрегатный флаг).

**Честный водораздел (подтверждено сверкой с кодом 25.06):** часть «популярных» форм
заблокирована доступом REST на `projectkap` — строить нечем без внутрисуточной телеметрии/
событий: **Объём топлива** (график уровня — нет intraday-телеметрии), **Заправки/сливы по
событиям** и **Журнал** (REST 404), **полные События** (events пуст), **Мультимедиа** (видео
вне контура). Эти НЕ строим, помечены ниже. **Стоянки/Движение** — строятся из архива треков
(`fact_track`), но он пока скудный (бэкфилл на трикле) → реализуем по мере наполнения.

## Очередь реализации

- **P1 (популярные + дыры, делаем первыми):**
  1. **Объём топлива** — график уровня топлива по ТС (данные есть, нужен ряд+рендер).
  2. **Заправки и сливы** — форма событий заправок/сливов (мульти-ТС, xls).
  3. **Местоположение** — fleet-wide карта живых позиций (поверх `get_vehicle_state`/архива).
  4. **Нарушения** — расширить детекцию (стоянки/обороты/ДУТ) сверх скоростного режима.
  5. **Посещение геозон** — табличная форма визитов (данные уже тянем).
- **P2 (детализация по ТС/сменам):** Движение, Стоянки, Работа группы, Посменный, События
  (События/Журнал — упираются в права REST; запросить учётку с провиженингом событий).
- **P3 (ниша/блок доступа/вне контура):** Выдача топлива, АТЗ-выдачи, Журнал (сырьё), Мультимедиа.

## Сверх паритета — наш value-add (в Omnicomm нет)

- **Учёт автошин по пробегу** (02.07): в `/acreports` Omnicomm отдельной формы нет (max — TPMS
  давление, только с датчиком). Реализовано у нас как зеркало Контроля ТО: ресурс комплекта
  копится с даты установки из годового архива, статусы + **износ в ₸**, подтверждение замены →
  сброс цикла. Вкладка «Шины», `GET /api/tyres` (+`/replace|plan|history`). MVP = комплект на ТС;
  позиции/оси/ротация — фаза 2. Движок `omnicomm_report/tyres.py`, стор `api/tyre_store.py`.

## Сквозные требования (из аудита — добавить ко всем нашим отчётам)
- **Экспорт XLS** на каждую табличную форму (у нас есть `/api/dashboard.xlsx` — распространить).
- **Печать/PDF** табличных форм.
- **Группы колонок** (вкладки внутри отчёта) как в «Сводном»/«Работе группы»/«Стоянках».
- **Пресеты периода** `сегодня/вчера/7д/месяц` + произвольный — у нас есть в `SyncBar`, унифицировать.
- **«Зафиксировать» (снапшот)** — у нас уже снапшот-кэш, можно дать кнопку фиксации в UI.
- **Рабочие столы** (карта+отчёт рядом) — паттерн Omnicomm, рассмотреть как пресет-композиции.

## Замечания по доступности (НЕ переоткрывать)
- `events/{id}` и «Журнал» по REST на `projectkap` закрыты (404/пусто) — полные «События»/«Журнал»
  требуют учётки с правами; на текущей учётке держим эвристики (геозон-визиты, агрегаты).
- Видео/фото (Мультимедиа) — отдельный контур, в Omnicomm у КАП не передаётся.
- Часть тяжёлых форм («Посещение геозон», «Посменный») в самом Omnicomm строятся долго на
  большой выборке — у нас это снимается снапшот-кэшем (считаем синком, отдаём мгновенно).

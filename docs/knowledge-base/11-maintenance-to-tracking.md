# Контроль ТО (планового техобслуживания) — спецификация фичи

> Продаваемая способность платформы: авто-учёт наработки и **заблаговременные алерты на плановое ТО** по моточасам/пробегу, с подтверждением выполнения. Из аудио Дмитрия 22.06.
> Связано с [10 Sensor Health](10-sensor-health-detection.md) (источник наработки), [06 §5](06-omnicomm-online-capabilities.md) (у Omnicomm свой «Контроль ТО»), [05 Рекомендации](05-recommendation-engine.md) (формат алёртов/эскалации).

## 1. Идея (со слов Дмитрия)
- Выдавать **алерты по наработке**: «техника наработала X моточасов (или Y км) → рекомендуем плановое ТО». В виде алёрта/строки в отчёте.
- Опираться на **их регламент** (сколько единиц можно одновременно отправить на ТО) — ИЛИ просто давать рекомендации, не заморачиваясь планированием.
- Алерты должны **«засыпать»** через время, ИЛИ дать **ответственному (диспетчеру)** управлять: подтвердил «ТО пройдено» → система **обновляет цикл** (отсчёт заново от этой даты).
- **Не дублировать** Omnicomm: там ТО уже ведётся (но вручную); дважды записывать времени нет.

## 2. Ключевая проблема: «контроль ТО от нуля»
Исторических данных о пройденных ТО у нас нет (звонить и просить показания со штатного счётчика в кабине по 2000 единиц — нереально). Решение Дмитрия: **стартуем от нуля** на момент подключения к платформе (T0) и считаем наработку из фида.

```
наработка(ТС) с T0:
    моточасы  = Σ engine_hours из фида (mv.worked / CAN моточасы)
    пробег    = Σ mileage из фида (mv.mileage / CAN одометр)   ← CAN упрощает (точный одометр)
ТО_наступает когда: наработка ≥ интервал_ТО(категория/модель)
предупреждать заблаговременно: «осталось N моточасов / M км» («Напомнить за», как у Omnicomm)
```

- **Интервал ТО** на единицу — справочник по категории/модели. Источник: согласовать с заказчиком, либо взять типовую периодичность из открытых источников, заполнить дефолтами.
- **CAN упрощает:** при наличии CAN брать пробег/моточасы с шины (точно), иначе аккумулировать из суточных `mv.worked`/`mv.mileage`. Контролировать можно **по моточасам И/ИЛИ по пробегу** (что раньше наступит).

## 3. Цикл подтверждения (критично — «как узнать, что ТО выполнено»)
Алерт «ТО через 50 моточасов / 100 км» бесполезен без обратной связи. Нужен **ответственный** (диспетчер/специалист по ТО) с правом:
```
[наработка ≥ порог] → алерт «Ожидается ТО для ТС X»
   → ответственный подтверждает «ТО пройдено (дата)»
   → система сбрасывает счётчик, новый цикл от даты подтверждения
   → алерт уходит/засыпает
```
Без подтверждения — алерт **«засыпает»** (snooze) и всплывает ближе к/при достижении порога, чтобы не спамить. Это требует: состояние ТО на ТС (хранилище) + UI-действие в портале + журнал подтверждений (кто/когда).

## 4. Не дублировать Omnicomm — решение по источнику
У Omnicomm есть «Контроль ТО» (по дате/пробегу/моточасам, событие «Ожидается ТО» — [06 §5](06-omnicomm-online-capabilities.md)), **но он требует ручного заведения заданий**. Два пути:
| Путь | Плюс | Минус |
|---|---|---|
| **A. Вести ТО у нас** (авто от нуля + подтверждение) | авто-наработка из фида, не нужно ручное ведение, своя логика подтверждения/засыпания | счётчик «от нуля», не знает прошлых ТО |
| B. Тянуть задания ТО из Omnicomm через API | единый источник | требует ручного ведения в Omnicomm (которое и так не делают), и доступность через API под вопросом |
**Рекомендация:** путь A (владеть ТО в платформе) — он закрывает реальную боль (авто-учёт без ручного ведения), Omnicomm-модуль ТО они всё равно вручную не заполняют. Дублирования нет, т.к. Omnicomm-ТО де-факто не используется.

## 5. Модель данных (эскиз)
```
to_plan:        ТС → {интервал_моточасы, интервал_км, напомнить_за}
to_state:       ТС → {наработка_моточасы_с_T0, наработка_км_с_T0, T0, статус}
to_confirm:     ТС → [{дата_подтверждения, кем, тип_ТО, наработка_на_момент}]  (журнал/история)
```
Наработка обновляется из фида при каждом запуске; статус ∈ {ok, приближается, ожидается, просрочено}.

## 6. Связь с продуктом
- Это ещё одна линия **алёртов/рекомендаций** в формате [05](05-recommendation-engine.md) и по модели ЦМ [07](07-monitoring-center-operations.md) (приоритет, засыпание, эскалация ответственному).
- **Upsell:** контроль ТО — отдельная продаваемая ценность (сервис «не пропусти плановое ТО по всему парку»). При этом честно: точность «от нуля» растёт по мере накопления; для прошлых ТО нужны их данные.
- Стыковка с [10](10-sensor-health-detection.md): моточасы/пробег берём из того же фида; при сбое CAN/датчика наработка по этому ТС неполна → пометить.

## 7. Открытые вопросы
- 🟠 Регламент ТО заказчика (интервалы по категориям/моделям; сколько единиц можно одновременно на ТО).
- 🟠 Считать по моточасам, по пробегу или min из двух — согласовать по типам техники.
- 🟠 Кто «ответственный» на стороне ДЗО (подтверждает ТО) — нужна роль/право в портале (в рамках scope ДЗО, [holding-product-requirements](../holding-product-requirements.md)).
- 🟠 Брать ли пробег/моточасы из CAN (если параметры передаются) — зависит от оснащения ТС.

# Развёртывание holding-платформы + настройка частоты обновления (R2.4)

Прод-стек: **Python-движки → FastAPI (`api/`) → Next.js статика (`web/out/`)**, за
реверс-прокси `deploy/holding_proxy.py` (один порт фронтит и статику, и `/api/*`).
Keep-alive — cron + flock (НЕ systemd). Подробности инфраструктуры — в рабочей
памяти проекта; здесь — то, что относится к ТЗ.

## Частота обновления данных (R2.4 — «сколько раз в сутки тянуть данные»)

### Инкрементальный синк (по умолчанию, каждые 3 часа)

**Принцип:** каждые 3ч довозим ТОЛЬКО свежие сутки в сырое хранилище (`api/raw_store.py`,
`data/cache/raw.db`) и пересобираем снимок из НАКОПЛЕННОГО за окно (30 сут), НЕ
перезабирая историю из Omnicomm. Это снимает зависание длинных периодов и резко
снижает нагрузку на сервер Omnicomm (тянем ~2 суток, а не 30).

Действующая cron-строка:
```
0 */3 * * * cd /home/ubuntu/omnicomm-holding-platform && flock -n /tmp/holding_sync.lock \
  curl -s --max-time 3600 -X POST 127.0.0.1:8810/api/sync/incremental \
  -H 'Content-Type: application/json' -d '{}' >> logs/sync.log 2>&1
```
Параметры (`config.py`, тело можно переопределить): `INGEST_WINDOW_DAYS=2` (сколько
свежих суток довозим), `VIEW_WINDOW_DAYS=30` (окно снимка из хранилища).

**Разовый backfill истории** (чтобы 30-дн окно сразу было полным):
```
curl -s --max-time 3600 -X POST 127.0.0.1:8810/api/sync/incremental \
  -d '{"ingest_days":30}'        # ~15-25 мин на медленной копии, чанками, с прогрессом
```
Покрытие хранилища: `python3 -c "from api import raw_store; print(raw_store.coverage())"`.
Ретеншн: `raw_store.prune_before(<unix>)` удаляет старше cutoff (агрегаты + визиты + треки).

### Локальный архив GPS-треков за год (мгновенная карточка ТС)

Требование: система держит у себя весь год телеметрии, открытие карточки ТС за любой
прошлый день — **мгновенно из локального архива** (`raw_store.fact_track`), в Omnicomm
при клике не ходим. Забор устроен бережно к серверу Omnicomm (`api/track_backfill.py`):
выделенный медленный лимит (`TRACK_BACKFILL_RATE_PER_MIN=40`/мин ≪ 170), тянем только
дни с движением (по агрегату), резюмируемо (уже добранные сутки пропускаются),
с кап-таймаутом на слайс. Хранится упрощённой полилинией (Дуглас-Пекер).

Порядок заполнения:
```
# 1) сперва агрегаты за год (нужны как индекс «какие ТС×сутки ехали»):
curl -s --max-time 3600 -X POST 127.0.0.1:8810/api/sync/incremental -d '{"ingest_days":365}'
# 2) затем треки — годовой бэкфилл короткими слайсами (резюмируется, гонять до заполнения):
curl -s --max-time 2000 -X POST 127.0.0.1:8810/api/track/backfill -d '{"days":365}'
```
Покрытие: `GET /api/track/coverage` (или `raw_store.track_coverage()`).

Действующие cron-строки (рядом с инкрементальным синком):
```
# ночной до-вод свежих треков (когда год залит — почти всё пропускается, дёшево)
30 2 * * * cd /home/ubuntu/omnicomm-holding-platform && flock -n /tmp/holding_track.lock \
  curl -s --max-time 2000 -X POST 127.0.0.1:8810/api/track/backfill \
  -H 'Content-Type: application/json' -d '{"days":2}' >> logs/track.log 2>&1
```
На время первичного годового залива можно временно поставить тот же крон с `{"days":365}`
почаще (напр. `0 */2 * * *`) — слайсы добивают остаток; вернуть на `{"days":2}` после
полного покрытия. Бэкфилл одиночный (single-flight) и капается под медленным лимитом —
дневной синк он не вытесняет (оба под общим аккаунт-лимитером ≤170/мин).

### Полный синк (по запросу / произвольный период)

`POST /api/sync` (`{demo, start_ts, end_ts}`) — собирает снимок за период «с нуля»
(используется кнопкой «Собрать»/произвольным периодом в портале). Чанкинг по
`REPORT_WINDOW_DAYS=3` сут не даёт виснуть; таймаут масштабируется под длину.

Чтения дашборда (`/api/dashboard`, `/api/sensor-health`, …) **никогда** не ходят в
Omnicomm — отдают снапшот из кэша мгновенно. Платформа = единственный буфер к
Omnicomm: раздача учёток ДЗО НЕ множит нагрузку (ДЗО читают из кэша). Глобальный
rate-limiter `≤170/мин на аккаунт` (`rate_limit.py`) — под лимит Omnicomm 180/мин.

## Что обновлять при выкатке кода

- **Бэкенд (`api/`, `src/`):** `git pull` → перезапустить uvicorn по PID (не `pkill -f`).
- **Фронт (`web/`):** `rm -rf out .next && pnpm build` → прокси раздаёт `web/out/` с
  диска (рестарт прокси не нужен).
- После выкатки — один `POST /api/sync` для тёплого кэша с новыми секциями
  (Sensor Health / Контроль ТО появляются только в снапшотах нового синка).

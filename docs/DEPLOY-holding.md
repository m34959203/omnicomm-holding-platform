# Развёртывание holding-платформы + настройка частоты обновления (R2.4)

Прод-стек: **Python-движки → FastAPI (`api/`) → Next.js статика (`web/out/`)**, за
реверс-прокси `deploy/holding_proxy.py` (один порт фронтит и статику, и `/api/*`).
Keep-alive — cron + flock (НЕ systemd). Подробности инфраструктуры — в рабочей
памяти проекта; здесь — то, что относится к ТЗ.

## Частота обновления данных (R2.4 — «сколько раз в сутки тянуть данные»)

Это **админ-настройка** (не пользовательская): задаётся в двух местах и согласована
между собой.

1. **`ENV REFRESH_TIMES_PER_DAY`** (`.env`, по умолчанию `8`) — декларация частоты;
   читается `config.REFRESH_TIMES_PER_DAY`, выводится в админ-сводке/доке.
2. **Крон-расписание синка** — фактический триггер тёплого кэша:

   | Частота | cron-выражение для `POST /api/sync` |
   |---|---|
   | 8×/сутки (по умолчанию) | `0 */3 * * *` |
   | 4×/сутки | `0 */6 * * *` |
   | 24×/сутки (ежечасно) | `0 * * * *` |
   | 48×/сутки (каждые 30 мин) | `*/30 * * * *` |

   Строка синка (flock-guard, без дублей):
   ```
   <cron> cd <repo> && flock -n /tmp/holding_sync.lock \
     curl -fsS -X POST 127.0.0.1:8810/api/sync -H 'Content-Type: application/json' \
     -d '{"demo":false}' >> logs/sync.log 2>&1
   ```

Чтения дашборда (`/api/dashboard`, `/api/sensor-health`, …) **никогда** не ходят в
Omnicomm — они отдают готовый снапшот из кэша мгновенно. Частота влияет только на
свежесть снапшота, не на скорость интерфейса.

> Изменение частоты = поменять `REFRESH_TIMES_PER_DAY` в `.env` **и** cron-строку
> `holding_sync.lock`. Рестарт API не требуется (значение читается на чтении).

## Что обновлять при выкатке кода

- **Бэкенд (`api/`, `src/`):** `git pull` → перезапустить uvicorn по PID (не `pkill -f`).
- **Фронт (`web/`):** `rm -rf out .next && pnpm build` → прокси раздаёт `web/out/` с
  диска (рестарт прокси не нужен).
- После выкатки — один `POST /api/sync` для тёплого кэша с новыми секциями
  (Sensor Health / Контроль ТО появляются только в снапшотах нового синка).

# Аудит: тип ТС → допустимая скорость в геозоне

> Дата: 2026-07-01. Автор: платформа (по запросу Дмитрия + аудио директора по развитию).
> Вопрос: «у каждого ТС в геозоне — своя допустимая скорость по его типу»
> (north-star `geozone_limit(геозона, категория ТС) → max_speed`, см. `knowledge-base/04`, `09`).

Все цифры получены с живой копии КАП (`projectkap`, 2026-07-01), не из догадок.

---

## 1. Требование (со слов директора, аудио 22.06 + 01.07)

- Транспорт делится на **типы** (легковая, бульдозер, самосвал, грузопассажирский, автобус, буровая…).
- Для **каждой геозоны** задаётся лимит скорости **для конкретного типа ТС**.
- Тип **уже пишут в ИМЯ ТС** в Omnicomm, но **«не во всех моделях»** — процесс идёт.
- Привязка «тип × геозона → лимит» **ещё не финализирована** («надо будет обсудить»).

## 2. Что реально есть в Omnicomm сегодня (сверено по API + докам)

| Сущность | Где лежит | Покрытие | Вывод |
|---|---|---|---|
| **Тип ТС** | в **имени ТС** (слово в конце: «…Легковая», «…Тягач», «…Пикап») | **507/1828 ТС = 27 %** | структурного поля НЕТ; парсим из имени |
| **Лимит геозоны** | в **имени геозоны** («…50 км/ч», «огр 30») | **384/401 = 95 %** | источник истины (kb-09), уже парсим |
| **Лимит по типу × геозоне** | **нигде** | 0 % | в API одна цифра на зону, без разреза по типу |
| Категория/профиль ТС (структурно) | `/ls/api/v1/vehicles` | — | **403** (нет прав на копе) |
| `terminal_type` | дерево ТС | 100 % | это **железо** (FAS…), НЕ тип кузова |

**Главный вывод:** и тип ТС, и лимит зоны Omnicomm держит **в текстовых именах**, а не в структурированных полях. Разреза «лимит для типа X в зоне Y» в API **нет** — его задаёт НАША модель (kb-04/09) либо будущая таблица КАП.

## 3. Достижимое покрытие типизации (1828 ТС; 177 не-ТС отфильтрованы `classify.is_transport`)

| Источник категории | ТС | Доля | Уверенность |
|---|---|---|---|
| Тип-слово в имени (метка КАП) | 507 | 27 % | высокая (сами проставили) |
| + Марочная эвристика (Hilux→пикап, Equinox→лёгкий, Caravelle→автобус, Atlas Copco→буровая, КАМАЗ/SHACMAN→грузовик) | +682 | +37 % | средняя |
| = Автоклассификация итого | **1189** | **~64 %** | — |
| Остаток (override/добор брендов) | 639 | 35 % | ручное |

Остаток — это `ГАЗ H758302` (голый бренд без модели), редкая китайская спецтехника (`VAAX`, `SHANZHONG`, `SUNWARD`), `ПКУ-3…10` (похоже, стационарные — проверить, не ТС ли), `Komatsu` (добавить в словарь → экскаватор/бульдозер). Реально дотягивается до **~75–80 %** расширением словаря марок, дальше — override-файл.

## 4. Словарь типов в парке → категории kb-04 → профиль лимита

Частоты найденных типов (из имён): буровая 150, водовоз 80, автобус 40, пикап 31, трактор 29,
вахтовка 26, легковая 25, экскаватор 23, бульдозер 21, тягач 20, погрузчик 19, микроавтобус 14,
кран 13, бортовой 12, самосвал 9, грейдер 4, спецтехника 5, …

| Категория (kb-04) | Типы из имён | Профиль лимита |
|---|---|---|
| **Тяжёлая/спецтехника** | буровая, экскаватор, бульдозер, погрузчик, кран/автокран, грейдер, трактор, вездеход, манипулятор, спецтехника | техдороги/полигоны **5–50 км/ч** (зонная матрица) — главный риск износа |
| **Грузовые >3,5 т** | самосвал, тягач/седельный, бортовой, водовоз, грузовой | 60 НП / 70–90 вне; с прицепом строже |
| **Пассажирские/автобусы** | автобус, микроавтобус, вахтовка, грузопассажирский | 60 НП / 90 вне; усиленный DSM |
| **Легковые/лёгкий коммерч.** | легковая, пикап, внедорожники (Hilux/Prado/Equinox) | по ПДД + геозона |
| **Опасные грузы (ADR)** | топливозаправщик/АТЗ, цистерна (топливо/химия) | строгий; список опасных грузов — за **Данияром** |

Модель (kb-09): `лимит(ТС, зона) = min(ПДД(категория), лимит_имени_зоны, зонная_матрица(категория, зона_риска))`.
Разрез «по типу» даёт **ПДД(категория) + зонная матрица** — не имя зоны (оно одно на всех).

## 5. Риски и пробелы

1. **Тип покрыт на 27 %** и зависит от того, как КАП дозаполнит имена → нужна наша классификация как страховка, иначе 73 % ТС без типа = без разреза по типу.
2. **Парсинг из текста хрупок:** «Грузопас автобус», «Спецтехника (автогрейдер)», опечатки, лат/рус. Нужна нормализация + тесты + override.
3. **Привязка «тип × зона → лимит» не определена КАП.** Пока источник разреза — наша матрица kb-09 (6 зон × категории), не данные Omnicomm.
4. **5 % геозон** без лимита в имени (17 шт) → нужен фолбэк (зонная матрица/дефолт по типу зоны).
5. **Права:** структурный профиль ТС под 403 — если КАП начнёт класть тип в поле, а не в имя, понадобится доступ.
6. **Не-ТС в списке** (ПКУ/ДЭС/ёмкости) не должны получать категорию/лимит — вентиль `is_transport` уже есть, доревизовать ПКУ.
7. **R-INV (kb-05):** превышение по техдороге = дисциплинарка СТ КАП, НЕ штраф КоАП; не выдумывать суммы к водителю.

## 6. Вердикт готовности

- **Геозонная половина готова** (95 % лимитов из имён) — уже работает в `detect_from_visits`.
- **Типовая половина — на 27 %** из Omnicomm; до ~75–80 % дотягивается нашей классификацией сейчас, до 100 % — override.
- **Разрез «лимит по типу в зоне»** реализуем НАШЕЙ моделью kb-09 уже сейчас; данные Omnicomm по типу подключаются как высший источник, когда/если появятся структурно.

## 7. Рекомендация (staged)

1. **Категоризация ТС** — `category` на ТС из приоритета: `override-файл > тип-слово в имени (КАП) > марочная эвристика`. Плюс `data/vehicle_categories.json` для правок. Ревью присвоения на реальном парке до того, как оно погонит деньги/нарушения.
2. **`geozone_limit(имя_зоны, категория)`** — `min(ПДД, лимит_имени, зонная_матрица)`; фолбэк для 5 % зон без лимита.
3. **Перепроводка `detect_from_visits`** на пер-ТС категорию → у каждого ТС в зоне своя допустимая скорость по типу; вывод в «Нарушениях»/карточке ТС.
4. Когда КАП определит «тип × зона → лимит» — подключить как явный источник поверх матрицы.

**Вывод:** не ждать Omnicomm — владеть категорией у себя (пробрасываемый источник), запускать разрез по типу на нашей модели сейчас, метки КАП из имён использовать как высший авто-источник по мере их появления.
